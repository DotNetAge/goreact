    核心组件设计¶

    引言¶

    本技术文档围绕 GoReAct 的核心认知循环“思考-行动-观察-终止”进行系统化梳理，重点解析思考器(Thinker)、执行器(Actor)、观察器(Observer)与终止器(Terminator)四类组件的设计理念、接口定义、实现细节与协作方式。文档从职责边界、状态管理、数据流、错误处理、生命周期、配置与扩展点等维度展开，并通过类图、时序图与流程图帮助读者快速理解组件在真实代码中的映射关系与运行机制。

    项目结构¶

    GoReAct 将 ReAct 循环抽象为可插拔的引擎组件，通过统一的 PipelineContext 在各步骤间传递状态；引擎层负责编排 Thinker/Actor/Observer/Terminator 四个核心步进器，形成稳定的执行管线。

    核心组件¶

    本节对四个核心组件进行职责与接口层面的概览，后续章节将深入到实现细节与交互流程。

    • 思考器(Thinker)
    • 职责：意图识别、工具/技能编排、RAG 检索、基于历史决策下一步动作或判定完成。
    • 接口：Think(ctx) -> error
    • 执行器(Actor)
    • 职责：根据最后一步的 Action，路由到具体工具并执行，产出原始结果或错误，写入上下文总线供观察器消费。
    • 接口：Act(ctx) -> error
    • 观察器(Observer)
    • 职责：将 Actor 的原始输出/错误转化为结构化、降噪后的自然语言反馈，注入当前 Trace 的 Observation 字段。
    • 接口：Observe(ctx) -> error
    • 终止器(Terminator)
    • 职责：在每次循环末尾评估是否应结束（成功、失败阈值、最大步数、资源耗尽、停滞检测）。
    • 接口：CheckTermination(ctx) -> (bool, error)

    架构总览¶

    ReActor 的执行循环由 Reactor 编排，按 Think → Act → Observe → CheckFinish 的顺序迭代，直到终止器判定结束。每一步均通过 PipelineContext 共享状态，组件之间通过接口解耦，支持可插拔替换与扩展。

    详细组件分析¶

    思考器(Thinker) 设计与实现¶

    • 设计理念
    • 将 LLM 的推理过程模块化为“意图解析→工具发现→提示合成→LLM 推理→模式化解析”的子流程，支持多种模式（如计划/规格/默认 ReAct），并内置自修复反射。
    • 关键接口与职责
    • Think(ctx): 决策下一步动作或标记完成；失败仅在推理过程本身致命时返回。
    • 内部状态管理
    • 通过 PipelineContext 累积 Traces、PlanSteps、CurrentPlan、附件、令牌用量与流式回调等。
    • 数据传递机制
    • 使用 FluentPromptBuilder 注入系统模板、任务、工具描述、历史对话与记忆片段；通过 LLM 客户端流式接收响应；解析后写回 Trace 或 FinalResult。
    • 错误处理策略
    • 工具列表枚举失败直接返回；LLM 调用失败透传；格式解析失败自动注入反思观测并继续循环。
    • 生命周期与配置
    • 支持 WithModel、WithToolManager、WithMemoryBank、WithSystemPrompt 等选项；默认模型、提示模板与简单工具管理器。
    • 扩展点
    • 可替换 LLM 客户端、提示模板、压缩策略、记忆模块与解析器；支持强制压缩命令触发更激进压缩。

    依赖分析¶

    • 组件耦合与内聚
    • 四个核心组件通过 PipelineContext 解耦，彼此仅依赖接口；Actor/Observer 依赖工具管理器；Thinker 依赖 LLM 客户端与提示构建器；Terminator 仅依赖上下文状态。
    • 外部依赖与集成点
    • Reactor 依赖 gochat 的 pipeline 库；Thinker 依赖提示构建、压缩、计数与解析器；Actor 依赖工具管理器；Observer 依赖上下文总线；Terminator 依赖上下文状态。
    • 潜在循环依赖
    • 未见直接循环依赖；各组件通过接口与上下文单向传递数据。
    • 接口契约与实现细节
    • Thinker/Actor/Observer/Terminator 均通过步骤包装器注入到 Pipeline；Reactor 负责编排与钩子注册。

    性能考虑¶

    • Token 与成本控制
    • Thinker 使用滑动窗口压缩策略与令牌计数器，限制提示长度；默认最大输出令牌上限；通过使用回调累加 Prompt/Completion 令牌。
    • 延迟与可观测性
    • Thinker 记录推理时延；各工具执行记录 latency 指标；上下文提供全局指标收集器。
    • 流式输出
    • Thinker 支持 LLM 流式事件回调，结合上下文钩子实现实时思想流输出。
    • 循环节流
    • Reactor 主循环在每次迭代后短暂休眠，降低 CPU 占用。

    故障排查指南¶

    • 常见问题定位
    • 思考器解析失败：检查提示模板与解析器；查看是否触发了格式错误反射观测。
    • 工具未注册：执行器会记录告警并写入错误；确认工具管理器已正确注册目标工具。
    • 观察器无输出：确认执行器是否写入 rawoutput/rawerror；检查上下文总线是否被提前清理。
    • 循环卡住：终止器会发出停滞警告；检查 Action 名称是否一致且输入未变化。
    • 日志与指标
    • 使用上下文日志器与指标收集器定位瓶颈；关注 Thinker 时延、各工具执行时延与令牌用量。
    • 重启与恢复
    • 若出现致命规则引擎错误，Reactor 会在主循环中设置完成原因并返回；可在上层捕获并决定重试策略。

    结论¶

    GoReAct 的核心组件以清晰的接口与上下文驱动实现了 ReAct 循环的可插拔架构：思考器负责认知决策，执行器负责物理/数字世界交互，观察器负责语义化反馈，终止器负责安全边界。通过 PipelineContext 统一承载状态、指标与钩子，组件间低耦合、高内聚，便于扩展与演进。建议在生产环境中结合安全钩子、资源监控与规则引擎扩展，进一步提升鲁棒性与可控性。
