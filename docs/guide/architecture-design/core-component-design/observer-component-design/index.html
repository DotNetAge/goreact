    观察器组件设计¶

    简介¶

    本文件围绕观察器（Observer）组件展开，系统阐述其在 ReAct 循环中的“观察与收集”职责，包括结果处理、状态更新与反馈机制；解释 Observer 接口的设计理念（观察结果的标准化、上下文信息收集、状态同步策略）；梳理默认实现的数据处理流程、错误捕获与异常恢复；说明观察器与思考器、执行器之间的数据交互模式、结果验证与质量控制；并给出配置选项、性能优化与扩展策略，帮助读者快速理解并安全地定制观察器行为。

    项目结构¶

    观察器组件位于独立包中，配合步骤封装、引擎编排与核心类型定义共同构成 ReAct 循环的关键一环。关键文件分布如下： - 接口与默认实现：observer.go、defaultobserver.go - 步骤封装：stepobserver.go - 引擎编排：reactor.go - 核心类型与上下文：types.go、context.go - 其他步骤（对比参考）：stepactor.go、stepthinker.go、stepcheckfinish.go - 可观测性接口：observability.go - 示例与测试：defaultobservertest.go、agent_test.go

    核心组件¶

    • Observer 接口：定义“观察”职责，接收最近一次执行器动作的原始输出/错误，负责解析、去噪、摘要、截断、语义化，并将标准化后的观察结果附着到当前 Trace 的上下文中。
    • 默认观察器实现：提供基础文本解析、错误语义化、复杂结构序列化、上下文清理等能力。
    • 观察器步骤封装：将 Observer 注入到通用流水线框架中，作为固定的一环参与 ReAct 循环。
    • 核心类型与上下文：Observation、Trace、PipelineContext 提供状态存储、历史记忆与跨组件共享通道。
    • 引擎编排：Reactor 组装 Thinker、Actor、Observer、Terminator 四大组件，驱动循环执行。

    架构总览¶

    ReAct 循环由四个核心组件组成：思考器（Thinker）、执行器（Actor）、观察器（Observer）、终止器（Terminator）。观察器处于第三环，负责将执行器返回的原始结果转换为大模型可理解的上下文，并更新当前 Trace 的观察项，供终止器判断是否结束会话。

    详细组件分析¶

    Observer 接口设计原理¶

    • 角色定位：作为 ReAct 引擎的“感官”，桥接物理/数字世界的嘈杂反馈与大模型的认知上下文。
    • 职责边界：
    • 解析与提取关键信息；
    • 去噪、摘要、截断以节约 Token 窗口；
    • 将系统错误语义化为自然语言，辅助思考器反思；
    • 将完整构建的 Observation 附着到当前 Trace。
    • 设计要点：
    • 输入来自上下文总线（Context Bus）中的键值（如 rawoutput、rawerror）；
    • 输出标准化为 Observation，包含 Data（可直接喂给 LLM）、Raw（原始数据用于调试/多模态）、Error（错误对象）、IsSuccess（是否技术成功）。

    故障排查指南¶

    • 症状：观察器未产生任何 Observation
    • 检查是否存在 Action（无 Action 则直接返回）；
    • 检查上下文总线是否写入 rawoutput/rawerror；
    • 确认观察器步骤已正确加入流水线。
    • 症状：Observation.Data 为空或不可解析
    • 检查输出类型是否为 string/[]byte/可序列化结构；
    • 查看序列化失败时的兜底提示。
    • 症状：错误未被语义化
    • 确认 raw_error 是否存在且为 error 类型；
    • 检查观察器是否正确设置 IsSuccess=false 与 Error 字段。
    • 症状：重复消费执行结果
    • 确认观察器是否清理了 raw_* 键；
    • 检查是否在多个步骤中重复读取同一键。

    结论¶

    观察器在 ReAct 循环中承担“感知与翻译”的关键角色：将执行器的嘈杂反馈转化为结构化、可理解、可控制的上下文，支撑思考器做出更稳健的决策，并为终止器提供可靠的收敛信号。默认实现提供了稳健的基础能力，同时保持接口开放，便于按需扩展（如多模态、Schema 校验、动态阈值等）。通过合理的配置与优化，可在保证质量的同时提升整体性能与稳定性。
