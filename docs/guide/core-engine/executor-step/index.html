    执行器步骤¶

    简介¶

    本章节聚焦“执行器步骤”（Actor Step）在 ReAct 循环中的核心职责：将思维阶段生成的“思考-行动”决策转化为可执行的物理或数字环境交互。执行器步骤负责： - 解析 Action 对象（包含工具名与输入参数） - 通过工具管理器进行工具路由与选择 - 安全检查与授权（含人机协同） - 执行工具函数、记录指标、封装结果 - 处理异步与错误恢复，并将原始输出回传给后续观察者步骤

    该文档将从代码级视角拆解 Actor 步骤的实现细节，结合与工具系统、安全钩子、引擎管道的协作关系，给出最佳实践与排障建议。

    项目结构¶

    执行器步骤位于步骤层（steps），其背后由 Actor 接口与默认实现承担实际执行；工具系统提供工具注册、参数校验与安全等级；安全钩子在执行前进行高风险工具的人机授权；引擎 Reactor 组装整个 ReAct 管道。

    核心组件¶

    • 执行器步骤（actorStep）：将 Actor 接口包装为管道步骤，负责在执行前检查安全钩子注入的错误，若存在则直接返回，使管道进入错误处理流程。
    • 默认执行器（defaultActor）：从 PipelineContext 中读取最后一次 Trace 的 Action，进行工具路由、执行、结果回填与指标记录。
    • 工具接口与管理器（Tool/Manager）：定义工具契约、安全等级与执行签名；SimpleManager 提供内存注册表与检索能力。
    • 安全钩子（SecurityHook）：在执行前拦截高风险工具，支持会话白名单与长时记忆白名单，必要时触发人工授权。
    • 引擎 Reactor：装配 Thinker、Actor、Observer、Terminator 与 Hooks，驱动 ReAct 循环。
    • 上下文与类型（PipelineContext/Action/Observation/Trace）：承载状态、中间变量、日志、指标与执行轨迹。

    详细组件分析¶

    执行器步骤（actorStep）¶

    • 角色定位：作为管道步骤，负责将 Actor 的执行映射到 ReAct 流程中。
    • 关键行为：
    • 在执行前检查 PipelineContext 是否已由安全钩子注入错误（如拒绝或授权失败），若有则直接返回，交由管道错误钩子处理。
    • 否则调用 Actor.Act，将最后一条 Trace 的 Action 交给 Actor 执行。
    • 与安全钩子的协作：通过 PipelineContext.Error 字段实现“前置拦截”，避免执行高风险动作。
        flowchart TD
    Start(["进入 actorStep.Execute"]) --> CheckErr["检查 state.Error"]
    CheckErr --> |存在错误| ReturnErr["返回错误，触发OnStepError"]
    CheckErr --> |无错误| CallAct["调用 Actor.Act(ctx)"]
    CallAct --> End(["结束"])
    ReturnErr --> End

    依赖分析¶

    • 执行器步骤依赖 Actor 接口与 PipelineContext；Actor 默认实现依赖工具管理器与工具接口。
    • 安全钩子依赖工具管理器与记忆银行（可选），并在执行前拦截 actor 步骤。
    • 引擎 Reactor 将所有组件与 Hooks 组装为管道，驱动 ReAct 循环。

    性能考虑¶

    • 指标记录：defaultActor 在执行后记录工具耗时指标，便于识别慢工具与热点路径。
    • 超时与重试：Actor 接口注释强调应处理内部重试与超时边界，建议在工具实现中设置合理的超时与重试策略。
    • 工具路由：SimpleManager 为内存查找，复杂场景可替换为基于向量检索的语义匹配以减少无效路由开销。
    • 日志与可观测性：通过 PipelineContext.Logger 与 Metrics 提升问题定位效率，建议在关键路径增加采样日志。

    故障排查指南¶

    • 工具未注册/名称不匹配
    • 现象：执行器记录告警并设置空结果/错误。
    • 排查：确认工具是否已注册到 Manager；Action.Name 与工具 Name 是否一致。
    • 高风险工具被拒绝
    • 现象：PipelineContext.Error 设置为拒绝错误或 Approver 未配置导致的错误。
    • 排查：检查 Approver 是否配置；确认会话/长期白名单；必要时调整工具安全等级。
    • 执行错误未被捕获
    • 现象：工具执行失败但未影响流程。
    • 排查：确认 defaultActor 是否正确设置 raw_error；观察者是否正确读取并语义化。
    • 无 Action 可执行
    • 现象：执行器直接返回。
    • 排查：确认思维步骤是否正确生成 Action；检查 PipelineContext.Traces 是否有最新 Trace。

    结论¶

    执行器步骤在 ReAct 循环中扮演“物理/数字世界接口”的关键角色。它通过解析 Action、路由工具、执行并回填结果，将思维阶段的决策转化为可感知的环境变化。配合安全钩子的人机授权、工具系统的参数校验与安全等级、以及引擎的可观测性与循环控制，形成了一个可扩展、可审计、可优化的执行闭环。实践中建议： - 明确工具安全等级并启用安全钩子 - 使用 ExtractInput 保证参数类型安全 - 利用指标与日志快速定位瓶颈 - 在工具实现中合理设置超时与重试
