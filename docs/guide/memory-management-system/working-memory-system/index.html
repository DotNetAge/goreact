    工作记忆系统¶

    引言¶

    本文件围绕 GoReAct 的工作记忆系统展开，重点阐述“易失性短期记忆”的设计理念与实现机制，并深入解析 DefaultMemoryManager 如何管理会话状态与临时数据存储。文档还提供工作记忆的配置、使用与管理方法示例，解释其与思考器（Thinker）、执行器（Actor）的交互关系与数据流转过程，覆盖生命周期管理、内存优化与并发安全策略，给出常见问题的排障建议与性能优化最佳实践，兼顾初学者与资深开发者的理解需求。

    项目结构¶

    工作记忆系统位于 pkg/memory 目录，核心接口与默认实现如下： - 接口定义：WorkingMemory、SemanticMemory、MuscleMemory、MemoryBank - 默认实现：DefaultMemoryManager（KV 内存存储 + 文件持久化） - 与思考器集成：Thinker 通过 MemoryBank 注入记忆到提示词模板 - 与引擎集成：Reactor 作为 ReAct 循环的装配器，承载 Thinker/Actor/Observer/Terminator 四大组件

    核心组件¶

    • WorkingMemory（短期记忆）：记录会话上下文、临时状态与授权信息；支持按意图召回、按键存储/检索、权重更新（时间衰减预留）、压缩清理。
    • MemoryBank（记忆体聚合）：统一协调短期、长期与经验三类记忆；提供 Compress 压缩能力。
    • DefaultMemoryManager（默认实现）：基于内存 KV 存储，支持并发安全（读写锁），提供 JSON 文件持久化/加载。
    • Thinker（思考器）：在构建提示词时注入 MemoryBank 的 Working/Semantic 记忆，形成“思维注入”闭环。
    • Reactor（引擎）：装配四组件并驱动 ReAct 循环；工作记忆贯穿每个步骤的状态传递。

    架构总览¶

    工作记忆在 ReAct 循环中的位置与职责： - 初始化阶段：Reactor 构建 PipelineContext，注入日志与指标。 - 思考阶段：Thinker 从 MemoryBank 召回 Working/Semantic 记忆，拼装到系统提示词中。 - 执行阶段：Actor 基于思考结果调用工具；期间可更新短期记忆权重。 - 观察阶段：Observer 返回观察结果，供后续思考与决策。 - 终止阶段：Terminator 判断是否结束；必要时进行记忆压缩与持久化。

    详细组件分析¶

    DefaultMemoryManager：短期记忆的实现与管理¶

    • 设计理念
    • 易失性短期记忆：以会话为单位的 KV 存储，适合维护对话连贯性与临时约束。
    • 并发安全：使用读写锁保障多协程下的安全访问。
    • 生命周期：支持压缩（清理空值）、持久化（JSON 文件）与加载（从文件恢复）。
    • 关键方法
    • Store/Retrieve：按会话ID与键值对进行存取。
    • Recall：返回当前会话的全部状态（作为“上下文记忆”注入提示词）。
    • Update：预留权重更新（时间衰减等）接口，当前为占位实现。
    • Compress：清理无效键值，保持上下文精简。
    • Persist/Load：将会话状态序列化到磁盘或从磁盘恢复。
    • 并发与线程安全
    • 读多写少场景采用 RWMutex，读锁共享、写锁独占，避免阻塞读取。
    • 持久化阶段使用读锁即可（序列化只读），提高并发吞吐。
    • 文件持久化策略
    • 默认目录 ./memory，文件命名规则为 {sessionID}_memory.json。
    • 加载时若文件不存在视为正常（首次会话）。

    性能考量¶

    • 并发与锁粒度
    • 读多写少场景使用 RWMutex，读锁共享，写锁独占，减少写路径阻塞。
    • 持久化阶段使用读锁，避免序列化写入阻塞读取。
    • 内存占用优化
    • 定期调用 Compress 清理空值键，降低上下文体积。
    • 控制会话数量与最大步数，防止上下文膨胀。
    • I/O 与持久化
    • 按会话粒度持久化，避免全局扫描；文件命名规范，便于运维与清理。
    • 提示词长度控制
    • Thinker 使用滑动窗口压缩策略，结合 MemoryBank 的上下文注入，平衡信息密度与上下文长度。

    故障排查指南¶

    • 会话未找到
    • 现象：Persist 返回“会话未找到”
    • 排查：确认 sessionID 是否正确传入；检查 Load 是否已将会话数据恢复到内存。
    • JSON 序列化失败
    • 现象：Persist/Load 抛出错误
    • 排查：检查存储值是否包含不可序列化类型；必要时转换为可序列化结构。
    • 并发冲突导致的数据不一致
    • 现象：读取到部分更新状态
    • 排查：确认所有读写均通过 DefaultMemoryManager 的带锁方法；避免绕过封装直接访问内部 map。
    • 记忆未生效
    • 现象：提示词中未看到记忆注入
    • 排查：确认 Thinker 的 createPromptBuilder 已从 MemoryBank 获取 Working/Semantic 记忆并注入模板变量；检查 RecallContext 的返回值。
    • 会话状态丢失
    • 现象：重启后上下文为空
    • 排查：确认 Load 已正确读取磁盘文件；检查 persistPath 权限与文件存在性。

    结论¶

    DefaultMemoryManager 以简洁的 KV 存储实现了“易失性短期记忆”，通过 RWMutex 保障并发安全，并提供压缩与持久化能力，满足会话状态管理与临时数据存储需求。配合 MemoryBank 接口与 Thinker 的记忆注入机制，工作记忆在 ReAct 循环中发挥“思维锚定”的关键作用，既消除了事实类与操作类幻觉，又提升了任务执行效率。建议在生产环境中结合压缩策略、I/O 优化与可观测性监控，持续保障稳定性与性能。
