    记忆管理系统¶

    更新摘要¶

    变更内容 - 更新了DefaultMemoryBank的架构描述，从简单内存管理器转变为三模态记忆系统 - 新增了工作记忆、语义记忆、肌肉记忆的具体实现和接口定义 - 更新了记忆系统的三模态设计理念和交互关系 - 补充了新的代码示例和最佳实践

    简介¶

    本文件系统性阐述 GoReAct 记忆管理系统的"三模态记忆体系"：短期工作记忆（易失性）、语义记忆（检索增强）与肌肉记忆（基于经验的标准操作程序）。文档围绕 MemoryBank 接口抽象、默认实现、与思考器（Thinker）、执行器（Actor）以及知识编译器（Compiler）的交互关系展开，提供配置与使用指引、最佳实践、性能优化与存储策略，并给出常见问题与数据一致性保障建议。内容兼顾初学者与资深开发者，既强调概念映射，也提供代码级定位路径。

    项目结构¶

    与记忆系统直接相关的核心位置如下： - pkg/memory：定义三类记忆接口与默认实现 - pkg/thinker：思考器在推理阶段注入记忆变量 - pkg/agent：智能体持有专属 MemoryBank - pkg/skill：技能与经验蒸馏（肌肉记忆）的承载 - pkg/cache：通用内存缓存（可作为短期记忆的辅助） - examples：示例工程展示如何装配与运行

    核心组件¶

    • MemoryBank 接口：统一抽象三类记忆引擎，提供 Working()、Semantic()、Muscle() 方法，以及 Compress() 压缩功能。
    • WorkingMemory：短期会话记忆，KV 存储，支持权重更新与压缩。
    • SemanticMemory：语义知识库（RAG/GraphRAG），提供只读知识召回。
    • MuscleMemory：经验记忆（肌肉记忆），保存蒸馏后的 SOP 快捷路径。
    • DefaultMemoryBank：默认实现，基于三模态记忆系统，支持 JSON 持久化/加载。

    详细组件分析¶

    MemoryBank 接口与三模态记忆¶

    • 设计理念：短期工作记忆（易失性）用于会话上下文与临时约束；语义记忆（检索增强）提供背景知识；肌肉记忆（经验）沉淀最佳实践。
    • 接口职责：
    • Working()：获取工作记忆实例
    • Semantic()：获取语义记忆实例
    • Muscle()：获取肌肉记忆实例
    • Compress()：压缩或修剪旧的记忆

    依赖分析¶

    • 组件耦合与内聚
    • Thinker 对 MemoryBank 松耦合（通过接口），可通过不同实现替换（如将 DefaultMemoryBank 替换为 RAG 支撑的高级实现）。
    • Agent 仅持有 MemoryBank 接口，职责清晰，利于测试替身与扩展。
    • Compiler 依赖 MemoryBank 的 MuscleMemory 接口，形成"经验沉淀—技能精炼"的闭环。
    • 外部依赖
    • Thinker 依赖提示词构建器、上下文压缩器、Token 计数器等工具模块。
    • DefaultMemoryBank 依赖标准库的 JSON、文件系统与同步原语。

    性能考量¶

    • 并发与锁粒度
    • DefaultMemoryBank 使用读写锁，读多写少场景下可提升吞吐；注意避免长时间持锁。
    • 序列化与 IO
    • Persist/Load 使用 JSON，建议在高频场景下考虑压缩或异步落盘，减少阻塞。
    • 记忆压缩
    • Compress 清理空值键，可在会话结束或定期清理时调用，降低上下文体积。
    • 缓存辅助
    • 可结合通用内存缓存（MemoryCache）作为短期工作记忆的热点缓存，配合 TTL/LRU 策略，减轻 KV 存储压力。
    • 参考：pkg/cache/memory.go:23-75、pkg/cache/memory.go:148-188

    故障排查指南¶

    • 记忆未生效
    • 检查是否正确将 MemoryBank 注入到 Thinker（参考：pkg/thinker/default_thinker.go:44）。
    • 确认提示词构建阶段已将记忆变量注入（参考：pkg/thinker/default_thinker.go:186-196）。
    • 会话状态为空
    • Recall 返回空可能表示会话不存在或为空；检查 Store/Load 是否正常（参考：pkg/memory/manager.go:107-123、pkg/memory/manager.go:125-145）。
    • 权重更新无效
    • Update 为占位实现，需替换为具备权重衰减与增益的高级实现（参考：pkg/memory/manager.go:76-91、pkg/memory/README.md:62-68）。
    • 蒸馏未触发
    • 确认 Terminator 是否在成功路径后触发 Compiler（参考：pkg/skill/compiler.go:11-18）。
    • 数据一致性
    • 使用读写锁保护并发访问；必要时在关键路径增加幂等写入与版本校验（建议）。

    结论¶

    GoReAct 的记忆系统以 MemoryBank 为核心抽象，将短期工作记忆、语义知识库与肌肉记忆有机整合，形成"事实—上下文—经验"的三模态闭环。通过在思考器中注入记忆、在执行器与终结器中沉淀经验，系统实现了从"开卷说明书"到"老司机"的认知跃迁。默认实现提供了可替换的起点，结合缓存与持久化策略，可在保证一致性的同时获得良好性能。

    附录¶

    • 最佳实践
    • 将 MemoryBank 作为 Agent 的必需组件，贯穿 ReAct 循环。
    • 优先使用语义记忆提供背景知识，再结合短期记忆与肌肉记忆优化路径。
    • 定期压缩与持久化短期记忆，避免上下文膨胀。
    • 将 Update 与 Compress 作为短期记忆的治理手段，配合权重衰减策略。
    • 性能优化
    • 使用 MemoryCache 作为短期热点缓存，设置合理 TTL 与容量。
    • 异步落盘与批量压缩，降低 IO 压力。
    • 在高并发场景下，尽量缩短持锁时间，避免跨网络调用在临界区内执行。
    • 存储策略
    • 短期记忆：内存 KV + 定期持久化（JSON）。
    • 语义记忆：对接 RAG/GraphRAG，按需检索与缓存。
    • 肌肉记忆：结构化 SOP 文档，支持版本化与精炼。
    • 扩展指南
    • 可以替换 DefaultMemoryBank 为支持真实 RAG 的实现。
    • 可以扩展 WorkingMemory 支持权重衰减算法。
    • 可以实现更复杂的 MuscleMemory 以支持编译态执行图。
