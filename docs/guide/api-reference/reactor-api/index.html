    Reactor API¶

    简介¶

    本文件为 Reactor API 的权威技术文档，面向希望在系统中集成 ReAct 思维循环（思考-行动-观察-终止）引擎的开发者与架构师。文档覆盖以下关键内容： - Reactor 结构体及其公共方法：NewReactor 构造函数、Run 方法 - 配置选项函数：WithEngineLogger、WithEngineMetrics、WithPipelineHook、WithThinker、WithActor、WithObserver、WithTerminator - 初始化流程、配置项作用与组合方式 - 错误处理策略与常见问题解决方案 - 性能优化建议与典型使用场景

    项目结构¶

    Reactor 所在模块位于引擎层，围绕“认知四元组”（Thinker、Actor、Observer、Terminator）构建可插拔的执行流水线，并通过 Pipeline Hook 与上下文可观测性（日志、指标）进行扩展。

    核心组件¶

    本节聚焦 Reactor 的公共 API 与关键数据结构，帮助快速理解其职责与用法。

    • Reactor 结构体

    • 字段概览

      • logger: 核心日志器（实现 core.Logger）
      • metrics: 核心指标器（实现 core.Metrics）
      • thinker: 思考组件（实现 thinker.Thinker）
      • actor: 行动组件（实现 actor.Actor）
      • observer: 观察组件（实现 observer.Observer）
      • terminator: 终止组件（实现 terminator.Terminator）
      • hooks: 流水线钩子列表（pipeline.Hook[*core.PipelineContext]）

    • 配置选项类型

    • Option: 函数式选项类型 func(*Reactor)，用于在构造阶段注入组件或行为

    • 公共方法

    • NewReactor(opts ...Option) *Reactor
      • 功能：创建并返回一个新的 Reactor 实例
      • 默认行为：设置默认日志器与指标器
      • 参数：可变数量的 Option
      • 返回：*Reactor
      • 复杂度：O(n)，n 为传入的 Option 数量
    • Run(ctx context.Context, sessionID, input string, customOpts ...core.ContextOption) (*core.PipelineContext, error)
      • 功能：执行一次完整的 ReAct 循环（Think → Act → Observe → CheckFinish），直至终止条件满足
      • 参数：
      • ctx：控制执行生命周期与取消
      • sessionID：会话标识
      • input：用户输入
      • customOpts：自定义上下文选项（如 WithLogger、WithMetrics、WithMaxSteps 等）
      • 返回：执行结果上下文与错误
      • 复杂度：取决于流水线步数与外部调用（LLM、工具等）

    架构总览¶

    Reactor 将四个核心组件组装为一个稳定的 ReAct 循环流水线，并支持全局日志与指标注入、流水线钩子扩展、以及上下文级可观测性配置。

    详细组件分析¶

    Reactor 构造与配置选项¶

    • NewReactor(opts ...Option) *Reactor
    • 默认注入：core.DefaultLogger()、core.DefaultMetrics()
    • 逐个应用传入的 Option，最终返回实例
    • 使用示例路径：examples/agent_test.go:52-60

    • 最佳实践：

      • 始终显式注入 Thinker、Actor、Observer、Terminator 四大组件
      • 如需统一日志与指标，优先使用 WithEngineLogger 与 WithEngineMetrics
      • 若需在执行前注入自定义上下文选项（如最大步数、附件、流式回调），通过 Run 的 customOpts 传入

    • 配置选项函数

    • WithEngineLogger(core.Logger) Option
      • 作用：为整个流水线注入全局日志器
      • 注意：Run 会在内部将该日志器注入到 PipelineContext 中
      • 示例路径：examples/agent_test.go:52-60
    • WithEngineMetrics(core.Metrics) Option
      • 作用：为整个流水线注入全局指标器
      • 注意：Run 会在内部将该指标器注入到 PipelineContext 中
      • 示例路径：examples/agent_test.go:52-60
    • WithPipelineHook(pipeline.Hook[*core.PipelineContext]) Option
      • 作用：注册流水线钩子，可在每次步骤前后执行自定义逻辑
      • 使用建议：用于审计、限流、状态持久化等横切关注点
    • WithThinker(thinker.Thinker) Option
      • 作用：注入自定义思考组件
      • 默认实现参考：pkg/thinker/default_thinker.go
    • WithActor(actor.Actor) Option
      • 作用：注入自定义行动组件
      • 默认实现参考：pkg/actor/default_actor.go
    • WithObserver(observer.Observer) Option
      • 作用：注入自定义观察组件
      • 默认实现参考：pkg/observer/default_observer.go

    • WithTerminator(terminator.Terminator) Option

      • 作用：注入自定义终止组件
      • 默认实现参考：pkg/terminator/default_terminator.go

    • 初始化与验证

    • Run 在开始执行前会校验四大组件是否均已配置，否则返回明确错误
    • Run 会将全局日志器与指标器注入到 PipelineContext，并合并自定义上下文选项

    • Run 内部构建 Pipeline[*]，按顺序添加 Thinker、Actor、Observer、CheckFinish 步骤，并注册所有钩子

    • 错误处理

    • 组件内部错误：若任一步骤返回错误，Run 会设置完成原因并返回上下文与错误
    • 组件缺失：若四大组件未全部配置，直接返回错误

    • 终止条件：由 Terminator 判定；默认实现包含最大步数与简单停滞检测

    • 使用示例

    • 完整示例（本地 Ollama/Qwen 与云端 DashScope）：
      • examples/agent_test.go:52-82
      • examples/agent_test.go:108-137

    故障排查指南¶

    • 常见错误与处理
    • “组件未配置”：当 Thinker/Actor/Observer/Terminator 任一缺失时，Run 返回明确错误。请确保通过相应 WithXxx 注入
      • 参考路径：reactor.go:113-115
    • “管道致命错误”：任一步骤返回错误时，Run 设置完成原因并返回上下文与错误
      • 参考路径：reactor.go:138-142
    • “最大步数限制”：默认 Terminator 达到 MaxSteps 后强制终止，并设置完成原因
      • 参考路径：default_terminator.go:29-35
    • “停滞检测”：默认 Terminator 对连续相同动作进行警告，必要时可结合业务规则增强
      • 参考路径：default_terminator.go:37-52
    • 单元测试参考
    • 通过 mock 组件验证循环次数、完成原因、最终结果与轨迹数量
      • 参考路径：pkg/engine/reactor_test.go:117-164

    结论¶

    Reactor 通过函数式选项与清晰的接口设计，提供了高度可插拔的 ReAct 引擎。借助 PipelineContext 的上下文选项与全局可观测性注入，开发者可以在不修改核心逻辑的前提下，灵活扩展日志、指标、钩子与终止策略。配合默认实现与示例工程，可快速搭建从本地推理到云端模型的智能代理系统。
