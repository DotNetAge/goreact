    Thinker API¶

    简介¶

    本文件为 Thinker API 的全面文档，覆盖接口定义、默认实现、提示协议、模式解析、配置选项、与 Reactor 的集成方式、错误处理策略以及性能优化建议。Thinker 是 ReAct 引擎的“大脑”，负责意图识别、工具/技能编排、RAG 检索以及基于执行历史做出下一步动作决策。默认实现包含内部子管线：意图与模式解析、工具发现、提示合成、上下文压缩、LLM 执行、输出反思与解析。

    项目结构¶

    • Thinker 接口与默认实现位于 pkg/thinker
    • 提示协议模板位于 pkg/thinker/prompt
    • 输出解析器位于 pkg/thinker/parser
    • 与 Reactor 的集成步骤位于 pkg/steps
    • 引擎主循环位于 pkg/engine
    • 核心类型与上下文定义位于 pkg/core
    • 示例位于 examples，展示如何将 Thinker 集成到 Reactor 中

    核心组件¶

    • Thinker 接口：定义单次思考入口 Think(ctx) error，用于在 ReAct 循环中评估当前状态并决定下一步动作或完成条件。
    • 默认 Thinker 实现：提供完整的内部子管线，包括模式解析、工具发现、提示合成、上下文压缩、LLM 执行与输出解析。
    • 输出解析器：从 LLM 原始文本中提取 Thought/Action/ActionInput/FinalAnswer，并在格式错误时进行自动反思。
    • 提示协议模板：包含 ReAct、规划与规格说明三种系统提示模板。
    • 集成步骤：将 Thinker 包装为管道步骤，供 Reactor 主循环调用。
    • 引擎：组装 Thinker、Actor、Observer、Terminator 四大组件，驱动 ReAct 循环。

    架构总览¶

    Thinker 在 ReAct 循环中的职责与位置如下：

        sequenceDiagram

    participant Parser as "输出解析器"

    Reactor->>Thinker : Think(ctx)

    LLM-->>Thinker : 流式响应片段
    Thinker->>Parser : processOutput(mode, raw)
    Parser-->>Thinker : Trace/FinalAnswer/完成标记
    Thinker-->>Reactor : 更新 ctx追加 Trace/完成
    Reactor-->>User : 返回最终结果

    详细组件分析¶

    Thinker 接口与默认实现¶

    • 接口职责
    • Think(ctx): 在 ReAct 循环中评估当前状态，决定：
      • 新的动作（附带“思考”理由作为新 Trace 追加到 PipelineContext）
      • 任务完成或失败（设置 ctx.IsFinished=true 并填充 ctx.FinalResult）

    • 错误返回：仅在思考过程本身致命失败时返回（如 LLM 网络崩溃、提示生成失败）

    • 默认实现关键流程

    • 自维护指令短路：如 /clear 清空上下文并结束；/compress 触发激进上下文压缩
    • 意图与模式解析：/plan 规划模式、/specs 规格模式、默认 ReAct 模式
    • 工具发现：通过工具管理器列出可用工具
    • 提示合成：使用 FluentPromptBuilder 组装系统模板、任务、工具描述、历史与记忆
    • 上下文压缩：滑动窗口策略，必要时强制压缩
    • LLM 执行：流式聊天，实时回调 OnThoughtStream

    • 输出解析：按模式解析 Thought/Action/ActionInput 或 FinalAnswer；格式错误时自动反思

    • 配置选项（Option 模式）

    • WithModel(model string)
    • WithToolManager(mgr tools.Manager)
    • WithMemoryBank(bank memory.MemoryBank)

    • WithSystemPrompt(tpl string)

    • 关键数据结构

    • PipelineContext：承载会话 ID、输入、步骤计数、完成标志、追踪、附件、计划、指标、日志、令牌用量、开始时间、错误、完成原因、思考流回调等
    • Action/Trace/Observation：标准化的动作、单轮迭代轨迹与观察结果

    故障排查指南¶

    • 常见错误与处理
    • 工具列表失败：工具管理器 ListAvailableTools 返回错误，需检查工具注册与权限
    • LLM 调用失败：网络异常或模型不可用，返回错误并由 Reactor 标记 PipelineFatalError
    • 输出解析失败：当无法解析 Thought/Action/ActionInput/FinalAnswer 时，自动追加 Format Error Reflection 观察，引导模型纠正
    • 上下文过长：启用 /compress 或调整压缩策略

    • 会话中断：/clear 可重置上下文，便于重新开始

    • 调试建议

    • 启用 OnThoughtStream 实时查看思考流
    • 查看 PipelineContext.Traces 与 LastTrace，定位问题环节
    • 检查 TokenAudit 与 FinishReason，确认完成状态与原因

    结论¶

    Thinker API 通过清晰的接口与强大的默认实现，提供了可扩展、可观察、可压缩的推理能力。结合 Reactor 的四组件循环，能够稳定地完成复杂任务。通过提示协议模板、输出解析器与上下文压缩策略，Thinker 在准确性、可解释性与性能之间取得良好平衡。建议在生产环境中结合流式思考、令牌审计与步数限制，以获得更好的用户体验与资源控制。

    API 定义与签名摘要¶

    • Thinker 接口
    • Think(ctx): 在 ReAct 循环中评估状态并决定下一步动作或完成条件
    • 默认 Thinker
    • Think(ctx): 完整内部子管线执行
    • resolveMode(input): 模式解析（/plan、/specs、默认 ReAct）
    • createPromptBuilder(ctx, tools, template): 提示合成
    • callLLM(ctx, messages, opts): 流式 LLM 调用
    • processOutput(ctx, mode, raw): 输出解析与反思
    • 输出解析器
    • ParseLLMOutput(raw): 解析 Thought/Action/ActionInput/FinalAnswer
    • ParsePlan(raw): 解析结构化计划
    • 提示协议
    • ReActSystemPrompt、PlanningSystemPrompt、SpecsSystemPrompt
    • 集成步骤
    • Thinker(t): 将 Thinker 包装为管道步骤
    • 引擎
    • NewReactor(opts...): 组装 Thinker/Actor/Observer/Terminator
    • Run(ctx, sessionID, input, customOpts...): 执行 ReAct 循环
