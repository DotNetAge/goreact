    可观测性类型¶

    简介¶

    本文件围绕 GoReAct 的可观测性类型与实现进行系统化说明，涵盖以下方面： - 观测性数据结构：日志接口与字段、指标接口与计数器、追踪与观察模型、令牌用量聚合等 - 实现方式：如何在 ReAct 系统中注入日志与指标钩子，如何在执行循环中采集与上报 - 集成点：引擎、步骤、观察者、上下文之间的可观测性协作 - 最佳实践：性能影响评估、配置建议、常见问题与调试技巧 - 示例路径：通过仓库中的测试与示例文件定位具体使用位置

    项目结构¶

    可观测性相关代码主要分布在如下模块： - 核心接口与上下文：日志、指标、令牌用量、执行上下文 - 日志实现：统一日志接口与基于 zap 的实现 - 指标实现：延迟、错误、成功、Token 消耗、资源使用等计数器与聚合 - 观察者：将执行结果标准化为可被 LLM 使用的观察文本 - 引擎与步骤：在 ReAct 循环中注入与传播可观测性钩子

    核心组件¶

    本节聚焦可观测性相关的数据结构与接口，以及它们在系统中的职责。

    • 日志接口与字段
    • 日志接口定义了 Debug/Info/Warn/Error 与 With 子日志能力，便于结构化记录
    • 字段类型支持字符串、整型、浮点、布尔、任意类型、错误、时长等

    • zap 实现提供生产与开发两种配置，统一字段转换

    • 指标接口与计数器

    • 指标接口定义延迟、错误、成功、Token 消耗、资源使用等记录方法
    • 默认实现包含计数器、定时器、Token 计数器、资源使用计数器

    • 提供聚合导出与重置能力

    • 执行上下文与追踪

    • 上下文携带会话 ID、输入、步骤、完成状态、最终结果、追踪列表、附件等
    • 观测性钩子（日志、指标）与令牌用量在上下文中注入与传播

    • 追踪记录单次循环的思考、动作与观察

    • 观察者与观察结果

    • 观察者负责将执行结果标准化为自然语言，附加到最近一次追踪

    • 观察结果包含清洗后的数据、原始输出、错误与是否成功标志

    • 令牌用量

    • 聚合提示词、补全与总 Token 数，提供安全累加方法

    架构总览¶

    ReAct 引擎在每次循环中依次执行思考、行动、观察与终止检查。可观测性通过上下文注入日志与指标钩子，并在观察阶段生成结构化观察结果，供后续步骤与外部系统消费。

    详细组件分析¶

    日志接口与实现¶

    • 接口设计
    • 支持 Debug/Info/Warn/Error 四级日志与 With 子日志能力
    • 字段类型覆盖常用场景，便于结构化输出
    • zap 实现
    • 提供生产与开发两种配置，时间键与编码器可定制
    • 将内部字段转换为 zap 字段，保证兼容性
    • 使用建议
    • 在引擎启动/结束、关键步骤前后记录 Info/Debug
    • 错误路径使用 Error 并附带关键字段
    • 使用 With 生成带标签的日志子句，提升可读性

    依赖关系分析¶

    • 接口与实现解耦
    • core 包提供 Logger/Metrics 接口与默认空实现，便于替换与禁用
    • log 与 metrics 分别提供接口与实现，保持低耦合
    • 上下文传播
    • Reactor 在 Run 中将日志与指标注入 PipelineContext，贯穿整个循环
    • 观察者与步骤
    • 观察步骤将 Observer 注入管道，形成稳定的执行链路

    性能考量¶

    • 日志开销
    • 结构化字段与 zap 编码器在 Info/Debug 层级应谨慎使用高频字段，避免过多字符串拼接
    • 开发环境可启用更详细的日志，生产环境建议降低日志级别
    • 指标开销
    • DefaultMetrics 使用互斥锁保护多计数器，聚合导出时复制引用以减少锁持有时间
    • 高频操作建议合并记录或批量导出，避免频繁 GetMetrics
    • 资源监控
    • ResourceMonitor 读取运行时内存与 GC 统计，注意采样频率与时间戳精度
    • 令牌用量
    • TokenUsage 提供 Add 方法进行安全累加，避免并发竞争
    • 配置建议
    • 在生产环境默认启用 DefaultMetrics 与 ZapLogger，必要时替换为自定义实现
    • 对于高吞吐场景，考虑异步导出或采样策略

    故障排查指南¶

    • 常见问题
    • 观察结果为空：确认是否存在 Action 尝试，检查上下文中的原始输出/错误键是否存在
    • 日志未输出：检查 WithLogger 是否正确注入，日志级别是否过低
    • 指标不更新：确认 Record* 方法调用路径，确保操作名一致
    • 调试技巧
    • 使用 WithThoughtStream 实时查看思考流，结合上下文令牌用量定位异常
    • 在观察者中记录 Debug 日志，验证错误语义化与输出序列化逻辑
    • 通过单元测试验证指标聚合行为，参考指标测试用例

    结论¶

    GoReAct 的可观测性通过清晰的接口与默认实现，将日志、指标与追踪无缝集成到 ReAct 循环中。核心要点包括： - 使用 Logger/Metrics 接口与 With* 注入选项，确保钩子在上下文中可用 - 观察者负责将原始执行结果转化为结构化观察，保障 LLM 输入质量 - 指标默认实现提供延迟、错误、成功、Token 与资源使用等多维度聚合 - 在生产环境中建议合理配置日志级别与指标采样，平衡可观测性与性能
