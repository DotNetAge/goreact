    核心接口定义¶

    引言¶

    本文件聚焦于 GoReAct 的核心接口定义与实现，系统阐述 Thinker（思考者）、Actor（执行者）、Observer（观察者）与 Terminator（终结者）四大接口的设计原则、方法签名、参数与返回值语义、协作关系与调用顺序，并结合默认实现与步骤包装器给出最佳实践、错误处理策略与性能优化建议。目标是帮助初学者快速上手，同时为有经验的开发者提供深入的技术参考。

    项目结构¶

    围绕 ReAct 循环引擎，核心由以下模块构成： - 接口层：thinker、actor、observer、terminator 四个包分别定义各自职责的接口 - 默认实现：每类接口均提供默认实现，便于直接使用或作为扩展基线 - 步骤适配：steps 包将接口封装为流水线步骤，注入到 Reactor 主循环 - 核心状态：core 包定义 PipelineContext、Trace、Action、Observation 等关键数据结构 - 引擎：engine 包的 Reactor 负责装配四类组件、注册钩子、驱动循环

    核心组件¶

    本节从接口职责、方法签名、参数与返回值、典型行为与约束等方面进行概述。

    • Thinker（思考者）
    • 职责：意图识别、工具/技能编排、RAG 检索、基于历史决策下一步动作或判定完成
    • 方法：Think(ctx *core.PipelineContext) error
    • 返回：仅当“思考过程本身致命失败”时返回错误；否则通过修改 ctx 决定继续或结束

    • 关键副作用：可能设置 ctx.IsFinished、填充 ctx.FinalResult、追加 Trace

    • Actor（执行者）

    • 职责：根据 Thinker 的 Action 执行具体工具或外部交互，捕获原始结果或错误
    • 方法：Act(ctx *core.PipelineContext) error
    • 返回：执行内部本地重试与超时边界；将结果写入 ctx 或直通给 Observer；错误通常不致命，而是通过上下文传递

    • 关键副作用：在 ctx 上设置 rawoutput/rawerror，供 Observer 使用

    • Observer（观察者）

    • 职责：将 Actor 的原始输出/错误转化为结构化认知反馈，供 LLM 反思
    • 方法：Observe(ctx *core.PipelineContext) error
    • 返回：解析/去噪/截断/语义化，最终将 *core.Observation 附加到当前 Trace

    • 关键副作用：清理上下文中的临时 bus 字段，避免污染后续步骤

    • Terminator（终结者）

    • 职责：安全地中断潜在无限/停滞迭代，成功/失败阈值、最大步数、资源耗尽等
    • 方法：CheckTermination(ctx *core.PipelineContext) (bool, error)
    • 返回：true 表示应终止（通常会设置 ctx.IsFinished 与 ctx.FinishReason）；err 表示规则引擎自身灾难性失败
    • 关键副作用：更新完成状态与原因

    架构总览¶

    ReActor 的主循环由 Reactor 驱动，按 Think → Act → Observe → CheckFinish 的顺序反复执行，直到 Terminator 判定终止。步骤包装器将接口适配为流水线步骤，Reactor 在运行期组装并执行。

    详细组件分析¶

    Thinker 接口与默认实现¶

    • 设计要点
    • Think 仅在“思考过程本身失败”时返回错误；通过 ctx 控制流程（继续/完成/切换输入）
    • 支持模式切换（如计划/规范/常规 ReAct），并可注入记忆与工具描述
    • 输出解析后决定是否结束任务并记录最终结果
    • 方法签名与语义
    • Think(ctx): 修改 ctx 的 IsFinished/FinalResult/Traces 等
    • 默认实现要点
    • 自维护命令短路（如清空上下文）
    • 工具发现、提示词合成、LLM 流式调用、输出解析与回退
    • 计数令牌用量、记录思考延迟、支持压缩策略

    依赖分析¶

    • Reactor 对四类组件的依赖通过函数式选项注入，确保装配清晰且可替换
    • 步骤包装器将接口与流水线解耦，便于扩展与测试
    • 默认实现之间无直接耦合，通过 PipelineContext 的共享状态进行协作
    • 核心类型（Action、Observation、Trace、PipelineContext）位于 core 包，被所有组件共享

    性能考量¶

    • 提示词压缩与滑动窗口策略：在默认 Thinker 中可启用更激进的压缩策略以控制上下文长度，降低令牌消耗
    • 指标记录：默认实现记录思考与工具执行延迟，便于定位瓶颈
    • 流式输出：LLM 流式响应可边生成边回调，减少端到端等待时间
    • 最大步数限制：防止无限循环导致资源耗尽
    • 观察者裁剪：对大体量输出进行序列化与截断，节省 LLM 上下文空间

    故障排查指南¶

    • 思考阶段错误
    • 现象：Think 返回致命错误，循环终止
    • 排查：检查 LLM 客户端连接、提示词构建、工具可用性
    • 参考路径：思考主流程:72-127
    • 执行阶段错误
    • 现象：Actor 报告工具不存在或执行失败
    • 排查：确认工具注册、权限、网络与超时；查看上下文 bus 中的 raw_error
    • 参考路径：执行与错误处理:38-77
    • 观察阶段错误
    • 现象：Observer 无法解析或语义化失败
    • 排查：检查原始输出类型与序列化；确认上下文 bus 是否正确清理
    • 参考路径：观察与语义化:27-90
    • 终止条件异常
    • 现象：循环未按预期结束或过早终止
    • 排查：核对 MaxSteps 设置、完成标志、停滞检测逻辑
    • 参考路径：终止条件判断:23-55
    • 循环卡顿
    • 现象：连续多次调用相同 Action
    • 排查：检查 Thinker 是否正确解析输出；必要时增强停滞检测
    • 参考路径：停滞检测:37-52

    结论¶

    GoReAct 的核心接口以清晰的职责分离与强一致的状态模型为基础，通过 Reactor 驱动的 ReAct 循环实现“思考—执行—观察—判定”的闭环。默认实现提供了开箱即用的能力，同时允许通过接口扩展与步骤包装器灵活定制。遵循本文的最佳实践与排错建议，可在保证稳定性的同时获得良好的性能与可维护性。
