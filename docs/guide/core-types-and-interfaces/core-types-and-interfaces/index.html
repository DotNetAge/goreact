    核心类型和接口¶

    简介¶

    本文件聚焦于 GoReAct 的核心类型与接口，系统性阐述以下内容： - 核心数据结构：PipelineContext、Trace、Action、Observation、TokenUsage 等的设计与职责 - 核心接口：Thinker、Actor、Observer、Terminator 的契约与实现关系 - 类型系统与各组件的交互流程：从“思考-行动-观察-终止”循环到上下文状态管理 - 实战示例路径：如何在引擎中装配组件并运行 ReAct 循环 - 最佳实践、性能优化与错误处理策略 - 常见类型转换与内存管理问题的解决方案

    项目结构¶

    GoReAct 将 ReAct 思维链流程拆分为四个核心角色（Thinker/Actor/Observer/Terminator），并通过一个统一的 PipelineContext 在各阶段传递状态与可观测性数据。引擎以管道化步骤驱动循环，确保可扩展、可测试与可监控。

    核心组件¶

    本节从“数据模型 + 接口契约 + 默认实现”的角度，全面解析核心类型与接口。

    • 数据模型
    • PipelineContext：承载会话标识、输入、当前步数、完成状态、最终结果、追踪列表、附件、计划步骤、最大步数、日志器、指标器、令牌用量、开始时间、错误原因、思考流回调以及内部状态字典等。
    • Trace：单次 ReAct 迭代的“思维-行动-观察”三元组，含步号、思考文本、动作与观察。
    • Action：由 Thinker 决策的动作，包含工具名与参数映射。
    • Observation：由 Observer 生成的标准化观察结果，包含清洗后的文本、原始输出、错误、是否成功标记。

    • TokenUsage：累计提示词与补全令牌用量。

    • 接口契约

    • Thinker：负责意图识别、工具/技能编排、RAG 检索与基于历史的下一步决策；可设置完成态与最终结果。
    • Actor：负责执行上一步 Action，映射到具体工具，捕获原始结果或错误，并写入上下文。
    • Observer：负责清洗 Actor 的原始输出，生成适合 LLM 的自然语言观察，必要时将系统错误语义化。

    • Terminator：负责安全边界检查，如达到最大步数、任务完成、停滞检测等，决定是否终止。

    • 默认实现

    • Thinker 默认实现：集成 LLM 客户端、工具管理、记忆、提示构建与压缩、流式思考输出、模式解析（ReAct/Plan/Specs）。
    • Actor 默认实现：通过工具管理器路由 Action 到具体工具，执行并写入上下文 bus。
    • Observer 默认实现：根据上下文 bus 中的 rawoutput/rawerror 生成 Observation，支持字符串、字节与结构体序列化。
    • Terminator 默认实现：检查完成态、最大步数与简单停滞检测。

    架构总览¶

    ReAct 循环由引擎驱动，按 Thinker → Actor → Observer → Terminator 的顺序执行，每轮结束后根据终止条件判断是否继续。上下文在各阶段共享，状态通过 AppendTrace 与 LastTrace 维护。

        sequenceDiagram
    participant E as "Reactor"
    participant TH as "Thinker"
    participant AC as "Actor"
    participant OB as "Observer"
    participant TM as "Terminator"
    loop 直到终止
    E->>TH : Think(ctx)
    TH-->>E : ctx(追加Trace/完成态)
    E->>AC : Act(ctx)
    AC-->>E : 写入上下文(raw_output/raw_error)
    E->>OB : Observe(ctx)
    OB-->>E : 附加Observation到Trace
    E->>TM : CheckTermination(ctx)
    TM-->>E : 是否终止
    end
    E-->>E : 返回最终PipelineContext

    详细组件分析¶

    数据模型：PipelineContext、Trace、Action、Observation、TokenUsage¶

    • PipelineContext
    • 关键字段：会话 ID、输入、当前步数、完成标志、最终结果、追踪列表、附件、计划步骤、当前计划索引、最大步数、日志器、指标器、令牌用量、开始时间、错误原因、思考流回调、内部状态字典。
    • 工具方法：AppendTrace、LastTrace、Get/Set、NewPipelineContext 及 ContextOption（附件、日志器、指标器、最大步数、思考流）。
    • 设计要点：以函数式选项注入依赖，避免构造函数膨胀；内部状态字典用于跨阶段传递非结构化数据。
    • Trace
    • 关键字段：步号、思考文本、Action 指针、Observation 指针。
    • 设计要点：作为短期记忆的“草稿板”，串联思考-行动-观察。
    • Action
    • 关键字段：工具名、参数映射。
    • 设计要点：与工具注册表解耦，便于动态路由。
    • Observation
    • 关键字段：清洗后的数据、原始输出、错误、是否成功。
    • 设计要点：区分“系统错误语义化”与“结构化数据序列化”，保证 LLM 可读性。
    • TokenUsage
    • 关键字段：提示词令牌、补全令牌、总令牌。
    • 工具方法：Add 累加。
    • 设计要点：集中统计，便于审计与成本控制。

    依赖分析¶

    • 组件内聚与耦合
    • Thinker 依赖 LLM 客户端、工具管理器、记忆模块、提示构建器与解析器，职责清晰但耦合度较高，适合通过选项注入替换实现。
    • Actor 仅依赖工具管理器与上下文，耦合度低，便于替换为异步/并发执行器。
    • Observer 仅依赖上下文 bus，耦合度最低，可独立演进。
    • Terminator 仅依赖上下文状态，逻辑简单，扩展点在于规则引擎。
    • 外部依赖
    • 日志与指标：通过 Logger/Metrics 接口抽象，默认实现为空操作，便于在生产环境替换。
    • LLM 客户端：通过 gochat 客户端抽象，Thinker 仅依赖其 ChatStream 与回调。
    • 循环依赖
    • 无直接循环依赖；引擎通过步骤封装 Thinker/Actor/Observer/Terminator，避免相互引用。

    性能考量¶

    • 令牌用量统计
    • 使用 TokenUsage 集中累加，便于成本控制与审计；建议在高并发场景下进行原子化更新或批量汇总。
    • 流式思考输出
    • Thinker 的流式回调 OnThoughtStream 可降低首字延迟，提升用户体验；注意避免在高频回调中做重 IO。
    • 观察清洗
    • 对大对象序列化为 JSON 时，建议在上游进行截断或摘要，减少上下文占用。
    • 执行度量
    • Actor 与 Thinker 分别记录工具与推理耗时，便于定位瓶颈；建议按工具维度聚合指标。
    • 步数与停滞
    • 设置合理 MaxSteps 防止无限循环；停滞检测可结合哈希比较输入映射以提高准确性。

    故障排查指南¶

    • 常见问题与定位
    • 工具未注册：Actor 会记录告警并写入错误，检查工具管理器注册与名称一致性。
    • LLM 输出格式不符：Thinker 的解析失败会自动注入反思观察，建议检查提示模板与输出约束。
    • 停滞循环：Terminator 发出告警，建议引入更严格的输入哈希对比与失败重试策略。
    • 令牌超限：通过提示压缩与历史截断缓解；关注 TokenUsage 累计值。
    • 错误处理与恢复
    • 上下文错误字段：Reactor 在管道致命错误时写入 FinishReason 与 Error，便于诊断。
    • 观察错误语义化：Observer 将系统错误转为自然语言，帮助 LLM 自修正。
    • 内存与资源
    • 上下文 bus：仅存放临时数据，Observer 完成后清理，避免长期持有。
    • 附件与历史：Traces 与 Attachments 会增长，建议在高并发场景下限制数量或周期性清理。

    结论¶

    GoReAct 的核心类型与接口以 PipelineContext 为中心，围绕“思考-行动-观察-终止”形成高内聚、低耦合的可扩展架构。通过接口抽象与默认实现分离，既满足快速落地，又支持深度定制。配合流式思考、令牌统计与停滞检测等机制，可在复杂任务中保持稳定与高效。

    附录¶

    • 最佳实践清单
    • 使用 ContextOption 注入日志器、指标器与最大步数，避免硬编码。
    • 在 Actor 层面统一处理超时与重试，向上游暴露稳定的 Observation。
    • 在 Thinker 层面严格约束输出格式，必要时增加二次校验与反射修正。
    • 在 Terminator 层面结合业务语义扩展终止规则（如成功率阈值、资源上限）。
    • 类型转换与内存管理
    • 将复杂结构体序列化为 JSON 字符串时，注意大小控制与异常分支。
    • 使用上下文 bus 传递中间结果，完成后及时清理，避免内存泄漏。
    • 对输入映射进行哈希比较以准确识别停滞，避免浅拷贝导致的误判。
