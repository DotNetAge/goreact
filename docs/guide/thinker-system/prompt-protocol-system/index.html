    提示协议系统¶

    简介¶

    本文件系统性地解析 GoReAct 的提示协议系统，重点覆盖以下方面： - Fluent API 设计理念与实现机制：提示模板构建、变量替换、链式调用、历史对话格式化与 Few-Shot 示例组织。 - 令牌计数器：tokenization 策略、计数准确性与缓存优化。 - 上下文压缩：滑动窗口、优先级保留、截断策略与混合策略。 - 提示格式化器：工具描述的多格式输出、参数 Schema 序列化与紧凑编码。 - 与思考器、工具系统的集成：如何在 ReAct 循环中合成提示并驱动推理。 - 常见优化问题与性能调优建议。

    项目结构¶

    提示协议系统位于 pkg/prompt 子目录，围绕“提示构建器”“压缩策略”“计数器”“格式化器”“提示模型”展开；同时在 pkg/thinker 与 pkg/engine 中体现与思考器、引擎的集成关系。

    核心组件¶

    • FluentPromptBuilder：以链式 API 组装系统提示与用户提示，支持模板变量、工具描述、历史对话、Few-Shot 示例与上下文压缩。
    • CompressionStrategy：提供截断、滑动窗口、优先级选择与混合策略，控制上下文长度与质量。
    • TokenCounter：提供简单估算、通用估算（按语言）与带缓存的计数器，支撑上下文压缩与提示长度控制。
    • ToolFormatter：提供多种工具描述格式（简单文本、JSON Schema、Markdown、紧凑格式），兼顾可读性与 token 节省。
    • Prompt/PromptManager：定义提示模型与模板管理器，支持注册、获取与渲染模板。

    架构总览¶

    提示协议系统在思考器内部被用于合成最终的系统与用户消息，随后由引擎驱动 ReAct 循环。工具列表与记忆内容通过变量注入到模板中，历史对话与 Few-Shot 示例参与上下文构建。

    详细组件分析¶

    FluentPromptBuilder：Fluent API 与提示合成¶

    • 设计要点
    • 链式调用：WithXxx 方法返回自身指针，便于连续配置。
    • 模板与变量：支持自定义系统/用户模板与变量注入，最终通过 text/template 渲染。
    • 历史与 Few-Shot：内置历史格式化器与示例拼接逻辑。
    • 上下文压缩：在 Build 前根据策略裁剪历史轮次，确保不超过 maxTokens。
    • 关键流程
    • prepareVariables：聚合自定义变量、工具、历史、示例与统计信息。
    • formatTools/formatHistory/formatFewShots：分别调用对应格式化器与拼接逻辑。
    • renderTemplate：使用模板引擎渲染系统/用户提示。
    • Build：先压缩历史，再渲染模板，最后封装为 Prompt。
        flowchart TD
    Start(["开始 Build"]) --> CheckComp["是否配置压缩策略且存在历史?"]
    CheckComp --> |否| PrepVars["准备变量"]
    CheckComp --> |是| Convert["转换 Turn 到压缩输入"]

    HasCustomSys --> |否| UseTpl["使用模板生成系统提示"]
    UseCustom --> Pack["封装为 Prompt"]
    UseTpl --> Pack
    Pack --> End(["结束"])

    依赖分析¶

    • 组件耦合
    • FluentPromptBuilder 依赖 ToolFormatter、HistoryFormatter、CompressionStrategy、TokenCounter。
    • defaultThinker 依赖 Builder、CompressionStrategy、TokenCounter、ToolFormatter，并与引擎/工具系统交互。
    • PromptManager 与 Prompt 作为轻量数据结构，被广泛复用。
    • 外部依赖
    • 模板渲染使用标准库 text/template。
    • 基准测试使用 testing 包。

    性能考虑¶

    • 计数器优化
    • 使用 CachedTokenCounter 缓存重复文本的计数结果，降低重复计算开销。
    • 在高频压缩与构建场景中，建议结合缓存提升整体性能。
    • 压缩策略选择
    • 短对话或高保真需求：优先使用优先级策略，保留 system 与近期轮次。
    • 长上下文：滑动窗口策略简单高效；若仍超限，可叠加截断策略。
    • 混合策略：按顺序应用，提前满足限制即停止，避免多余计算。
    • 工具描述格式
    • 长上下文场景优先 CompactFormatter，减少 token 占用。
    • 需要结构化参数时使用 JSONSchemaFormatter，但注意 token 成本。
    • 模板渲染
    • 尽量减少模板复杂度与变量数量，避免深层嵌套与重复渲染。

    故障排查指南¶

    • 提示过长导致截断
    • 现象：输出被截断或上下文丢失。
    • 排查：检查 maxTokens 设置、压缩策略与计数器准确性；必要时增大 maxTokens 或调整压缩强度。
    • 历史压缩误删关键信息
    • 现象：system 消息或近期关键轮次被移除。
    • 排查：确认优先级策略的保留项配置（如 KeepSystemFirst、KeepRecent）。
    • 工具描述格式不匹配
    • 现象：LLM 无法正确解析工具参数。
    • 排查：核对 ToolDesc/ParameterSchema 的字段完整性；必要时切换为 JSONSchemaFormatter。
    • 计数偏差
    • 现象：估算与实际 token 不一致。
    • 排查：使用 UniversalEstimator 并启用缓存；在关键路径上引入基准测试验证性能与准确性。

    结论¶

    GoReAct 的提示协议系统通过 Fluent API 将模板、变量、历史与工具整合为统一的提示构建流程，并借助多样的压缩策略与计数器保障上下文长度与质量。配合思考器与引擎的紧密协作，系统实现了从提示合成到 ReAct 推理的完整闭环。实践中应结合场景选择合适的压缩与格式化策略，并利用缓存与基准测试持续优化性能与稳定性。
