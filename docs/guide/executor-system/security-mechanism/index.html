    安全机制¶

    简介¶

    本文件系统化阐述 GoReact 的安全机制，重点围绕“安全钩子”（Security Hook）的设计原理与实现细节展开，覆盖以下主题： - 安全级别定义与工具分类 - 权限控制策略与人机协作授权（HITL） - 工具执行的安全监控、访问控制与审计日志 - 安全配置示例与最佳实践 - 安全钩子与工具执行器的集成方式、应用场景与故障处理 - 安全测试方法、漏洞防护策略与合规性考虑

    GoReact 将“安全”下沉到 Actor 层作为“原子级”的最后一道防线，通过 Pipeline Hook 在执行前对高风险工具进行拦截与授权，结合会话级白名单与长期记忆白名单实现“可撤销、可审计、可复用”的授权闭环。

    项目结构¶

    与安全机制直接相关的核心目录与文件如下： - 安全钩子与授权：pkg/actor/securityhook.go - 工具与安全级别：pkg/tools/tool.go、pkg/tools/builtin/*.go - 执行器与工具路由：pkg/actor/defaultactor.go - 上下文与追踪：pkg/core/context.go、pkg/core/types.go - 内存与白名单持久化：pkg/memory/manager.go - 引擎与管道：pkg/engine/reactor.go - 安全设计说明：pkg/actor/README.md

    核心组件¶

    • 安全钩子（SecurityHook）
    • 作用：在 Actor 执行前拦截高风险工具调用，触发人机协作授权流程（HITL），并在授权后自动写入会话白名单。
    • 关键行为：识别工具安全级别、检查会话/长期白名单、调用人机审批器、记录日志与错误。
    • 工具与安全级别（Tool 接口与内置工具）
    • 定义：工具必须声明安全级别（LevelSafe、LevelSensitive、LevelHighRisk），用于决定是否需要授权。
    • 示例：计算器（安全）、Bash（高风险）。
    • 执行器（Actor）
    • 作用：根据最后一步 Trace 中的动作，路由到具体工具并执行；在未挂载安全钩子时直接执行。
    • 上下文与追踪（PipelineContext、Trace、Action）
    • 作用：承载会话状态、追踪链路、中间变量（如 rawoutput/rawerror）、日志与指标。
    • 记忆与白名单（MemoryBank）
    • 作用：提供会话级与长期级白名单存储，支持授权后的自动白名单注入与持久化。

    架构总览¶

    安全钩子通过 Pipeline Hook 机制在 Actor 步骤前介入，依据工具安全级别与白名单策略决定是否阻塞等待人工授权。授权通过后，将工具名写入会话白名单，避免重复授权。

    详细组件分析¶

    安全钩子（SecurityHook）设计与实现¶

    • 触发条件
    • 仅在 Actor 步骤开始时生效
    • 当存在最近一次 Trace 且其中包含 Action
    • 工具安全级别达到高风险阈值（默认 LevelHighRisk）
    • 白名单策略
    • 会话级白名单：在 PipelineContext 中以键值形式存储，键格式为“whitelist:{toolName}”
    • 长期白名单：通过 MemoryBank.Working().Retrieve 查询，若命中则放行
    • 人机协作授权（HITL）
    • 若未配置审批器，则直接报错提示需要人工授权
    • 审批器接收上下文、工具与输入，返回批准与否
    • 批准后自动写入会话白名单，避免重复授权
    • 错误处理
    • 审批过程出错：记录授权失败错误
    • 用户拒绝：设置特定错误标识，便于上层区分

    依赖分析¶

    • 安全钩子依赖
    • 工具管理器：用于获取工具并读取安全级别
    • 记忆管理器：用于查询/写入长期白名单
    • 上下文：用于读取 Trace、写入白名单、记录日志与错误
    • 执行器依赖
    • 工具管理器：用于工具路由
    • 上下文：用于记录执行结果与错误
    • 引擎依赖
    • 管道：用于注册 Hook 并驱动循环

    性能考量¶

    • 安全钩子开销
    • 仅在 Actor 步骤开始时检查，且仅对高风险工具生效，通常为常数时间复杂度
    • 白名单查询为 O(1)，长期白名单查询为 O(1) 读取
    • 执行器开销
    • 工具路由与执行为 O(1)，延迟记录为微小额外开销
    • 指标与日志
    • 可通过指标收集器记录工具执行延迟、错误计数与资源使用，便于性能分析

    故障排查指南¶

    • 常见问题与定位
    • 高风险工具未触发授权：确认工具安全级别是否达到高风险阈值；确认是否正确注册安全钩子；确认审批器是否配置
    • 用户拒绝后仍报错：检查 PipelineContext.Error 是否为用户拒绝错误；确认上层是否按错误类型区分处理
    • 白名单未生效：检查会话白名单键格式；确认审批通过后是否写入；确认后续步骤是否命中白名单
    • 长期白名单未命中：确认 MemoryBank.Working().Retrieve 是否正确返回；确认持久化/加载流程
    • 单元测试参考
    • 安全钩子批准与白名单绕过：验证审批器调用与会话白名单写入
    • 用户拒绝：验证错误类型与 PipelineContext.Error 设置

    结论¶

    GoReact 的安全机制以“安全钩子 + 工具安全级别 + 人机协作授权 + 白名单策略”为核心，将安全下沉到 Actor 层作为最后一道防线。通过 Pipeline Hook 优雅地在不侵入执行器的情况下实现授权拦截，结合会话与长期白名单实现“可撤销、可审计、可复用”的授权闭环。配合完善的日志与指标体系，能够有效支撑安全监控、审计与合规需求。

    附录¶

    安全配置示例与最佳实践¶

    • 工具安全级别标注
    • 对所有注册工具显式声明安全级别，避免误判
    • 高风险工具（如 Bash、数据库写操作）必须标注为高风险
    • 安全钩子启用
    • 在组装引擎时注册安全钩子；未注册即视为授予 Agent 全权限（适用于受控环境）
    • 审批器设计
    • 审批器应具备幂等性与可审计性；建议记录审批请求、输入、决策与时间戳
    • 支持三种决策：拒绝、单次批准、批准并加入白名单
    • 白名单策略
    • 会话白名单：默认生效，适合短期信任场景
    • 长期白名单：通过 MemoryBank 持久化，适合已验证可信工具
    • 日志与审计
    • 记录高风险工具检测、授权请求、审批结果、拒绝原因
    • 结合 PipelineContext.Error 与日志字段，形成完整审计轨迹
