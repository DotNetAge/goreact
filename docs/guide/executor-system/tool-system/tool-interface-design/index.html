    工具接口设计¶

    简介¶

    本文件面向 GoReAct 工具接口设计，系统性阐述 Tool 接口的设计理念与方法定义，解释安全级别枚举及其使用场景，详解 MapTool 作为工具包装器的实现原理与用法，并给出 ExtractInput 的使用示例与最佳实践。文档同时结合内置工具实现，帮助读者快速理解如何在实际项目中正确、安全地扩展与使用工具接口。

    项目结构¶

    GoReAct 的工具接口位于 pkg/tools 目录，内置工具位于 pkg/tools/builtin。工具接口采用最小职责原则，通过统一的接口抽象不同能力，配合安全级别与管理器实现，形成可发现、可授权、可扩展的工具体系。

    核心组件¶

    • Tool 接口：定义工具的唯一标识、能力描述、安全级别与执行逻辑，统一了工具的调用契约。
    • 安全级别枚举：对工具进行风险分级，支撑授权与审计。
    • MapTool：将任意函数包装为 Tool，便于快速实现简单工具。
    • ExtractInput：将 map 输入安全地反序列化为严格结构体，降低类型不一致带来的错误。
    • 管理器接口：负责工具的注册、检索与上下文相关工具列表生成。

    架构总览¶

    工具接口围绕“最小接口 + 可选包装器 + 输入校验”的设计，形成如下交互流程：

        sequenceDiagram
    participant Caller as "调用方"
    participant Tool as "Tool 实现"
    participant Wrap as "MapTool"
    participant Util as "ExtractInput"
    Caller->>Tool : 调用 Execute(ctx, map)
    alt 使用 MapTool
    Tool->>Wrap : 委托执行
    Wrap->>Util : ExtractInput(输入, 目标结构体)
    Util-->>Wrap : 解析成功/失败
    Wrap->>Wrap : 调用内部 ExecuteFunc
    Wrap-->>Caller : 返回结果或错误
    else 直接实现 Tool
    Tool->>Util : ExtractInput(输入, 目标结构体)
    Util-->>Tool : 解析成功/失败
    Tool-->>Caller : 返回结果或错误
    end

    详细组件分析¶

    Tool 接口与方法定义¶

    • Name()：返回工具的唯一名称，用于思维链选择与调用。
    • Description()：清晰描述工具能力、输入输出，辅助系统提示词构建。
    • SecurityLevel()：声明工具的安全级别，用于授权与策略控制。
    • Execute(ctx, input)：执行工具逻辑，接收原始 map 输入，返回任意结果或错误。

    实现要点 - 输入为 map[string]any，来源于 LLM 的 JSON ActionInput；建议通过 ExtractInput 转换为结构体，提升类型安全。 - 输出为任意类型，便于内置工具返回统一的结构化结果。

    故障排查指南¶

    • 参数缺失或类型错误：使用 validateRequired/validateRequiredString 进行前置校验，确保关键字段存在且类型正确。
    • 文件安全问题：调用 validateFileSafety 防止访问受限文件或越权路径。
    • 除零与非法运算：在 Calculator 中对除法分母进行检查。
    • 文件过大或二进制文件：Read/Grep 已内置大小与类型过滤，避免不必要的 IO。
    • 命令执行失败：Bash 返回错误信息，结合日志与审计定位问题。

    结论¶

    GoReAct 的工具接口以简洁的契约与明确的安全级别为核心，辅以 MapTool 与 ExtractInput，既保证了易用性，又提升了类型安全与可维护性。内置工具覆盖文件操作、搜索与执行等核心场景，体现了“单一职责、组合强大”的设计思想。在生产环境中，建议结合授权策略、审计与资源限制，安全地启用高危工具，并通过测试用例验证工具行为。

    附录¶

    最佳实践清单¶

    • 明确定义工具的 Name/Description，确保系统提示词与用户理解一致。
    • 正确设置 SecurityLevel，配合 Actor 授权策略实施差异化控制。
    • 使用 ExtractInput 将 map 输入转换为结构体，减少类型判断与错误。
    • 对外部命令与文件写入等高风险操作，增加白名单、超时与审计。
    • 为工具编写单元测试，覆盖正常路径与边界条件。

    设计模式参考¶

    • 策略模式：通过 Tool 接口抽象不同工具策略，Actor 可统一调度。
    • 包装器模式：MapTool 将函数式逻辑包装为 Tool，降低实现成本。
    • 工厂模式：内置工具通过 NewXxx 工厂函数创建实例，集中初始化逻辑。
