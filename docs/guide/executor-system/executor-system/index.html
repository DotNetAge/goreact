    执行器系统¶

    简介¶

    本文件面向 GoReAct 执行器系统，围绕 Actor 接口与默认实现、工具系统、安全钩子与安全策略、以及与工具管理器的集成关系进行系统性说明。文档同时提供配置与工具调用的示例路径、自定义工具开发指南、测试策略与最佳实践，并给出常见安全问题与性能优化建议。

    项目结构¶

    执行器系统由“思维-行动-观察-终止”四步循环构成，核心位于引擎模块；执行器 Actor 负责将思维组件生成的动作映射到具体工具并执行；工具系统提供统一的工具接口与管理器；安全钩子在高风险工具执行前进行人机协作授权与白名单控制。

    核心组件¶

    • Actor 接口与默认实现：负责从上下文提取动作、路由到工具、执行并回填结果。
    • 工具系统：定义工具接口、安全等级、工具管理器与简单管理器实现。
    • 安全钩子：在高风险工具执行前进行人机协作授权与会话级白名单控制。
    • 引擎 Reactor：组装四步循环，注入日志、指标与管道钩子，驱动主循环。

    详细组件分析¶

    Actor 接口与默认实现¶

    • 接口职责：从上下文最后一条轨迹中读取动作，路由到工具，执行并回填原始输出与错误。
    • 默认实现要点：
    • 通过工具管理器按名称获取工具；若不存在，记录告警并回填空输出与错误。
    • 记录工具执行耗时指标。
    • 将执行结果或错误写入上下文键值，供后续观察者与终止条件使用。

    依赖关系分析¶

    • 组件耦合：
    • Actor 依赖工具管理器；工具管理器提供工具检索能力。
    • SecurityHook 同时依赖工具管理器与记忆银行，用于白名单与授权。
    • Reactor 组合四步组件与管道钩子，形成主循环。
    • 外部依赖：
    • 引擎使用通用管道框架进行步骤编排与钩子注册。
    • 工具执行可能涉及外部系统（如 Bash），应结合安全策略与沙箱限制。

    性能考虑¶

    • 指标记录：Actor 在执行工具时记录延迟指标，便于定位慢工具与瓶颈。
    • 超时与重试：Actor 层面应自行处理本地重试与超时边界，避免上层重复处理。
    • 工具选择：优先使用安全与敏感级别的工具，减少高风险工具的执行频率与范围。
    • 日志与可观测性：通过引擎注入的日志与指标，持续监控工具执行耗时与成功率。

    故障排查指南¶

    • 常见问题与定位：
    • 动作为空：Actor 在无动作时直接返回，检查思维组件是否正确生成动作。
    • 工具未注册：当工具不存在时，Actor 记录告警并回填错误；确认工具管理器注册情况。
    • 高风险工具被拒绝：检查授权器是否配置、会话白名单是否生效。
    • 授权失败：确认授权回调逻辑与上下文错误信息。
    • 单元测试参考：
    • Actor 行为验证：动作存在/不存在、错误处理。
    • 安全钩子授权/拒绝与白名单绕过。
    • 内置工具功能验证：计算器、Cron、Grep、Replace 等。

    结论¶

    执行器系统通过清晰的 Actor 接口与默认实现、可扩展的工具系统与安全钩子，实现了从“思考-行动-观察-终止”的稳健循环。配合工具管理器与安全策略，系统在保证灵活性的同时兼顾安全性与可观测性。建议在生产环境中严格配置授权器、启用长期白名单、并结合指标监控进行持续优化。
