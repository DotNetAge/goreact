    Actor 接口和实现¶

    简介¶

    本文件围绕 GoReAct 的 Actor 接口及其默认实现展开，系统阐述 Actor 的设计理念、职责边界、与 PipelineContext 的交互方式、工具调用机制、并发与重试策略、以及安全拦截与白名单机制。文档还提供可复用的配置选项、最佳实践与常见问题解决方案，帮助开发者快速实现自定义 Actor 并安全地集成到 ReAct 循环中。

    项目结构¶

    Actor 相关代码主要分布在以下模块： - 接口与默认实现：pkg/actor - 工具抽象与管理：pkg/tools - 执行引擎与步骤封装：pkg/engine、pkg/steps - 执行上下文与类型：pkg/core - 示例与测试：examples、pkg/actor/*_test.go

    核心组件¶

    • Actor 接口：定义 Act(ctx) 的职责，负责从 PipelineContext 中提取 Action 并执行对应工具，记录原始输出/错误，内部处理本地重试与超时边界。
    • 默认 Actor 实现：基于工具管理器进行工具路由与执行，记录延迟指标，将结果写回 PipelineContext。
    • 安全钩子 SecurityHook：在 Actor 步骤前拦截高风险工具调用，支持人工授权、单次授权与会话白名单。
    • 工具接口与管理器：统一抽象工具的名称、描述、安全等级与执行方法；SimpleManager 提供内存注册表与简单语义检索。
    • 执行引擎与步骤：Reactor 将 Thinker、Actor、Observer、Terminator 组装为流水线；Actor 步骤封装将 Actor 接口映射为流水线步骤。

    架构总览¶

    Actor 在 ReAct 循环中的位置与职责如下： - Thinker 输出 Action 与 ActionInput； - Actor 从 PipelineContext 最新 Trace 中读取 Action，通过工具管理器路由到具体工具； - 执行工具并记录原始输出/错误； - 将结果写回 PipelineContext，供 Observer 处理。

    详细组件分析¶

    Actor 接口设计与职责¶

    • 设计理念：Actor 是 ReAct 的“双手”，负责将 Thinker 的结构化决策转化为对外部环境的实际交互。
    • 关键职责：
    • 从 PipelineContext 的最新 Trace 中提取 Action；
    • 将 Action 名称映射到工具实现；
    • 执行工具并捕获原始结果或错误；
    • 内部处理本地重试与超时边界；
    • 将结果写回 PipelineContext，供后续步骤使用。

    故障排查指南¶

    • 常见问题与定位：
    • 工具未注册：默认 Actor 会记录警告并写入空结果/错误，检查工具管理器注册列表。
    • 高风险工具被拒绝：确认是否配置了审批器，或检查会话/长期白名单。
    • 上下文无 Action：确认 Thinker 是否正确生成 Action 并写入 Trace。
    • 单元测试参考：
    • 自定义 Actor 行为与错误处理可通过 mockActor 测试用例参考。
    • SecurityHook 的授权/拒绝逻辑可通过测试用例验证。

    结论¶

    Actor 作为 ReAct 的执行环节，承担着从结构化决策到真实外部交互的关键职责。默认实现提供了简洁可靠的工具路由与执行流程，并通过 Pipeline Hook 机制实现了可插拔的安全拦截。结合 PipelineContext 的可观测性与工具抽象，开发者可以快速实现自定义 Actor 并安全地集成到完整的认知循环中。

    如何实现自定义 Actor¶

    • 基本步骤：
    • 实现 Actor 接口的 Act(ctx) 方法；
    • 从 ctx.LastTrace() 读取 Action；
    • 通过工具管理器获取工具并执行；
    • 将 rawoutput/rawerror 写回上下文；
    • 可选：记录指标、处理错误、实现本地重试。
