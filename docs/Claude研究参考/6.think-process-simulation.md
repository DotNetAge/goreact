## 🧠 Claude Think 阶段内部推理过程模拟

### 输入
```
User: 帮我把这个 PR 的 PDF 报告转成 Markdown
```

### Step 1: 意图分析（Intent Recognition）
```
用户想要：
1. 处理 PDF 文件 ← 关键词触发！
2. 格式转换（PDF → Markdown）
3. 可能涉及 PR 相关操作
```

### Step 2: 扫描可用工具（Tool Schema Matching）

| 工具名 | 描述关键词匹配 | 决策 |
|--------|--------------|------|
| **Read** | "可以读取 PDF 文件" | ✅ 需要先读取 PDF |
| **Write** | "创建文件" | ✅ 可能需要写入 Markdown |
| **Bash** | "系统命令" | ❌ 有专用工具，不用 Bash |
| **Glob** | "搜索文件" | ❌ 不需要搜索 |
| **SkillTool** | "Execute a skill..." | ⚠️ 检查是否有相关技能 |

### Step 3: 扫描可用技能（Skill Listing Matching）

从 `<system-reminder>` 中看到的技能列表：
```
- pdf: PDF 文档处理 - 当用户需要处理、分析或转换PDF文档时
- commit: Git 提交助手 - 当需要提交代码时
- review: PR 代码审查 - 当需要审查PR时
```

**匹配过程**：
- 用户提到 "PDF" → 匹配 `pdf` 技能的 `whenToUse`: *"当用户需要处理、分析或转换PDF文档时"*
- 用户提到 "PR" → 可能也匹配 `review` 技能？
- 但主要任务是 **PDF 转换**，不是 PR 审查 → 主要匹配 `pdf`

### Step 4: 应用强制规则（Rule Application）

根据 SkillTool prompt：
> "When a skill matches the user's request, this is a **BLOCKING REQUIREMENT**: 
> invoke the relevant Skill tool **BEFORE** generating any other response"

✅ **决策：必须先调用 SkillTool！**

### Step 5: 构建工具调用

```json
{
  "type": "tool_use",
  "id": "toolu_xxx",
  "name": "SkillTool",
  "input": {
    "skill": "pdf",
    "args": "将 PR 的 PDF 报告转换为 Markdown 格式"
  }
}
```

### 为什么不直接用 Read + Write？

1. **`pdf` 技能可能包含专业的 PDF 解析逻辑**（提取文本、表格、图片）
2. **`whenToUse` 明确指出这是 PDF 处理的专业技能**
3. **System Prompt 要求优先使用专用能力**
4. **技能可能有特殊的参数处理或后处理流程**

### 替代场景：如果用户说 *"看看 report.pdf 里写了什么"*

这种情况下：
- 意图是**查看内容**，不是**转换格式**
- `Read` 工具的 prompt 说："This tool can read PDF files..."
- 可能直接调用 `Read("report.pdf")` 就够了
- 不需要调用 `pdf` 技能（除非需要复杂解析）
