## goreact (ReActor) 框架全面评估报告

### 一、整体架构评估

项目采用 T-A-O（Think-Act-Observe）循环模式，分层为 `core`（数据模型）→ `reactor`（引擎核心）→ `tools`（工具实现）→ 根包 `Agent`（门面）。架构方向正确，接口设计简洁（5 个 interface），包依赖方向单一无循环。但作为一个"可以跑起来"的 MVP，距离生产可用还有不少差距。

### 二、Critical 级问题

**1. `Agent.Ask()` 是空实现**
`agent.go:43-47`：`Ask()` 直接返回空字符串，完全没有调用 `reactor.Run()`。这意味着根包的 `Agent` 门面是个空壳，用户无法通过 `Agent` 使用框架。

**2. `pendingTasks` 是包级全局变量，不支持多 Reactor 实例隔离**
`task_tools.go:61-62`：`pendingTasks` 是 `var` 全局 map，如果用户创建多个 `Reactor` 实例，所有实例的 SubAgent 结果会混在同一个 map 中。应该将 pendingTasks 绑定到 `defaultReactor` 实例上。

**3. `read.go` 自定义 `max` 函数遮蔽 Go 1.21+ 内置函数**
`tools/read.go:142-147`：自定义了 `func max(a, b int) int`，而 Go 1.21+ 已将 `max` 作为内置泛型函数。虽然编译通过，但语义上可能产生混淆，且在未来 Go 版本中可能引发编译错误。

**4. `bash.go` 安全级别标记为 `LevelHighRisk` 但无实际安全检查**
`tools/bash.go` 中 `SecurityLevel` 标记为高风险，但 `reactor.go` 中的 `ToolRegistry.ExecuteTool()` 在执行工具前并未检查 `SecurityLevel`，安全标记形同虚设。更严重的是，`bash` 工具可以执行任意 shell 命令（包括 `rm -rf /`），而框架没有任何安全策略或权限控制层。

### 三、High 级问题

**5. `FileEditTool` 未实现 `core.FuncTool` 接口**
`tools/edit.go:14`：`NewFileEditTool()` 返回的是 `*FileEditTool` 而不是 `core.FuncTool`。虽然 Go 的隐式接口允许它作为 `core.FuncTool` 使用，但 `reactor.go:108` 中注册时调用的是 `NewFileEditTool()`（返回 `*FileEditTool`），而 `RegisterTool` 期望 `core.FuncTool`。这实际上是可行的（因为实现了接口），但与其他工具工厂函数返回 `core.FuncTool` 的风格不一致，降低了可读性。

**6. ConversationHistory 追加逻辑问题 — 角色语义错乱**
`reactor.go:455-465`：T-A-O 循环中，Think 的推理以 `assistant` 角色追加到 `ConversationHistory`，Observation 以 `user` 角色追加。这在语义上是反直觉的 — Observation（工具返回结果）被标记为 `user` 发送的消息。虽然对 LLM 来说可能不影响结果（因为 prompt 的 `<rules>` 不会被历史污染），但如果 LLM 的对话历史管理器对角色有假设（比如 OpenAI 的 system/user/assistant 严格约束），可能会导致 API 错误。

**7. `truncate` 函数按字节截断而非按字符**
`reactor/thought.go:138-143` 和 `reactor/reactor.go:439-441`：`s[:maxLen]` 是按字节截断，对中文等多字节字符会截断到半个字符，导致乱码。同样 `Observation.Result` 的 2000 字符截断也是按字节。

**8. `web_fetch.go` 无 SSRF 防护**
`tools/web_fetch.go:36-68`：接受任意 URL 参数，没有检查是否为内网地址（如 `http://127.0.0.1`、`http://metadata.google.internal` 等）。这是一个经典的 SSRF（Server-Side Request Forgery）漏洞。

**9. `email.go` 的 `setFlag` 中使用字符串字面量作为 `imap.Flag`**
`tools/email.go:503`：`flags := []any{flag}`，其中 `flag` 是 `string` 类型，但 `imap.Store` 期望的是 `imap.Flag`（`string` 类型别名）。虽然运行时可能正常工作，但这是类型不安全的用法。

**10. `todo_write` 工具是空实现**
`tools/todo.go:48-51`：无论传入什么参数都返回"Todo list updated successfully"，完全不解析参数也不存储状态。Batch/Architect skill 中引用了这个工具，但调用它毫无效果。

**11. `lsp` 工具是空实现**
`tools/lsp.go:51-65`：所有 LSP 操作（definition/references/symbols）只返回格式化字符串，不连接任何 LSP 服务器。

### 四、Medium 级问题

**12. `defaultReactor` 结构体的所有字段都是小写未导出，但 `NewReactor` 返回的是 `*defaultReactor`**
`reactor/reactor.go:79-120`：`NewReactor` 返回 `*defaultReactor`（未导出类型），这意味着用户无法在包外部直接持有 reactor 变量。虽然 `reactor.go` 中 `Run()` 方法挂在 `*defaultReactor` 上（可以被外部调用），但用户无法声明一个 `reactor.ReActor` 类型的变量并赋值给 `*defaultReactor`，因为 `ReActor` 是接口而 `defaultReactor` 未导出。这限制了可扩展性 — 用户无法创建自定义的 `ReActor` 实现（除非在 reactor 包内）。

**13. `classifyIntent` 每次调用都新建 `gochat.Client()`**
`reactor.go:148-179`：`callLLMWithHistory` 每次都 `gochat.Client().Config(...).Model(...)` 创建新 client。如果 `gochat.Client()` 不是单例或连接池，会导致频繁创建/销毁 HTTP 连接，影响性能。

**14. `InMemoryTaskManager` 的任务 ID 生成不可靠**
`core/task.go:59`：`id := fmt.Sprintf("task_%d", len(m.tasks)+1)` — 如果任务被删除（虽然当前没有删除接口），ID 会重复。应该用递增计数器或 UUID。

**15. `Observation` 缺少结构化错误传递**
`reactor/observation.go`：`NewErrorObservation` 只接受 `string` 类型的错误信息，丢失了原始 `error` 的类型信息和堆栈。而 `Action` 结构体中 `Error` 字段是 `error` 类型但 `ErrorMsg` 是 `string`，两种表示不一致。

**16. `SkillRegistry.FindApplicableSkills` 接受 `any` 类型参数**
`reactor/skill_registry.go:41`：`context any` 类型参数缺乏类型安全，内部通过 type assertion 转换为 `*Intent`。应该定义一个接口或在 `SkillRegistry` 接口上使用 `*Intent` 类型。

**17. `Replace` 工具缺少文件安全检查**
`tools/replace.go:33-47`：接受 `path` 参数但没有调用 `ValidateFileSafety()`，可以直接修改工作目录外的文件。

**18. `read.go` 的 `ValidateFileSafety` 与 `read` 工具参数名不匹配**
`tools/read.go:45`：读取工具参数名是 `path`，`tools/grep.go:60`：grep 搜索路径硬编码为 `.`。而 `tools/write.go` 和 `tools/replace.go` 也使用 `path`。但 `file_edit` 工具用的是 `file_path`。参数命名不一致。

**19. `core/prompts.go` 是空占位文件**
只有 `package core` 和一行空常量声明，应删除或补充内容。

**20. `core/memory.go` 接口过于简单且无实现**
只定义了 `Search` 和 `Save` 方法，没有内存实现，也没有在框架中被使用。

### 五、Low 级问题

**21. 中英文注释混用**：部分文件纯中文注释，部分纯英文，部分混合。建议统一。

**22. `gochat` 依赖使用本地 replace**：`go.mod` 中 `gochat` 是 `replace` 指令指向本地路径，不是正式的模块依赖。

**23. `reactor/reactor_test.go` 存在但工具测试更全面**：核心引擎的 T-A-O 循环缺乏端到端集成测试。

**24. `ContextWindow` 在 `Agent` 中声明但从未使用**：`agent.go:14`。

**25. `core/model.go` 的 `ToGochatModel` 方法未被使用**：框架中 LLM 调用完全在 `reactor.callLLMWithHistory` 中直接使用 `gochat.Client()` 链式 API，`ModelConfig` 的转换方法没有被调用。

### 六、可扩展性评估

**当前的限制：**

1. **无法自定义 ReActor 实现**：`ReActor` 接口在 reactor 包内定义但 `defaultReactor` 未导出，外部用户无法创建替代实现
2. **工具注册与 Reactor 生命周期耦合**：`NewReactor()` 内部硬编码注册了 15+ 工具，用户无法通过配置选择启用哪些工具
3. **Skill 只能通过 TriggerRules 匹配**：没有显式激活 skill 的 API，只能依赖 Intent 分类结果自动匹配
4. **没有插件系统**：不支持运行时动态加载工具或 skill
5. **没有中间件/hook 机制**：无法在 T-A-O 循环的各个阶段插入自定义逻辑（如日志、限流、审计）
6. **LLM 调用与框架紧耦合**：`callLLMWithHistory` 硬编码了 `gochat` 的使用方式，替换 LLM 后端需要修改 reactor 包代码

**建议的改进方向：**

- 将 `defaultReactor` 导出或提供 `Option` 模式让用户配置工具集
- 添加 `ReactorOption` 函数式选项模式
- 为 `ToolRegistry.ExecuteTool` 添加安全策略 hook
- 将 `callLLMWithHistory` 抽象为 `LLMCaller` 接口，支持不同 LLM 后端
- 为 T-A-O 循环各阶段添加事件 hook（`BeforeThink`、`AfterAct` 等）
- 将 `pendingTasks` 绑定到 reactor 实例而非包级变量

### 七、总结

| 维度 | 评分 | 说明 |
|---|---|---|
| 架构设计 | 7/10 | 分层清晰，接口简洁，但门面层（Agent）空实现 |
| 代码质量 | 6/10 | 部分工具空实现，类型安全性不足，截断逻辑有 bug |
| 安全性 | 4/10 | bash 无限制执行、SSRF 漏洞、安全级别未生效、文件路径检查不一致 |
| 可扩展性 | 5/10 | 无法自定义 ReActor，工具硬编码注册，无 hook 机制 |
| 测试覆盖 | 5/10 | 工具层测试较全，核心引擎缺乏集成测试 |
| 生产就绪度 | 3/10 | 多个空实现（Agent.Ask、todo_write、lsp），安全漏洞待修复 |

项目作为一个 ReAct 模式的 MVP 原型验证了核心概念（T-A-O 循环、意图分类、Skill 触发、SubAgent 调度），方向正确。如果要走向生产可用，最优先需要解决的是：Agent 门面实现、安全策略生效、按字符截断、修复 SSRF 漏洞，以及 pendingTasks 实例隔离。
