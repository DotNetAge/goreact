# 无限上下文（滑动窗口）实现方案

## 一、已确认的架构决策

### 决策 1：单层上下文管理（Reactor 层唯一）

- **Agent 层** = 纯门面模式，仅持有 Reactor、组合 Config/Model、提供 `Ask()` 入口
- **Reactor 层** = 唯一的上下文窗口管理者，持有 ContextWindow + SessionStore
- **后果**：当前 Agent 中 `ContextWindow` 的持有和管理代码需要**下沉到 Reactor**，Agent 不再直接操作上下文

```
调用链重构后：
Client → Agent.Ask() → Reactor.Run()
                      ↑
                 Reactor 内部：
                 ContextWindow (短期记忆)
                 SessionStore  (持久化层)
                 Memory/RAG    (长期记忆/知识)
```

### 决策 2：SessionStore 与 Memory 的职责边界

| 组件              | 定位              | 职责                                           | 类比                  |
| ----------------- | ----------------- | ---------------------------------------------- | --------------------- |
| **ContextWindow** | 短期记忆（RAM）   | 当前活跃的对话窗口，有 Token 上限              | 进程的工作内存        |
| **SessionStore**  | 持久化日志（WAL） | 有序、完整的会话历史存储与恢复                 | 数据库的预写日志      |
| **Memory / RAG**  | 长期知识库        | 从滑出消息中提取语义化知识，按 intent 反向注入 | 知识图谱 / 向量数据库 |

**数据流闭环**：

```
用户输入 → ContextWindow.AddMessage() → SessionStore.Append()
                                            ↓
                              [Token 使用率 ≥ 阈值?]
                                            ↓ 是
                              ContextWindow.Slide() 批量移除 oldest N 条
                                            ↓
                              被滑出的消息 → SessionStore.OnSlide() 回调/事件
                                            ↓
                        客户端/Memory 消费 → 语义化处理 → RAG 存储
                                            ↓
                        下次 Think 阶段 → Memory.Retrieve(intent) → 注入 Prompt
```

### 决策 3：滑动策略 — 基于 Token 比例的批量滑动

#### 研究结论：Token 预留比例

综合业界最佳实践（neurolink, Mindwave, getmaxim.ai, OpenAI 文档等）：

| 场景                      | 推荐预留比例             | 来源                                          |
| ------------------------- | ------------------------ | --------------------------------------------- |
| 简单对话（单轮问答）      | 20-25% 用于输出          | neurolink: `maxTokens * 0.8`                  |
| 复杂 Agent（T-A-O 循环）  | **30-40%** 用于输出+迭代 | T-A-O 每轮产生 Thought + Action + Observation |
| 安全缓冲区                | 10-15% 紧急容量          | getmaxim.ai 生产经验                          |
| "Lost in the Middle" 阈值 | 超过 50-70% 后质量下降   | Liu et al. 2023, Greg Kamradt 实验            |

#### GoReact 最终采用的滑动参数

```go
type SlideConfig struct {
    // SlideTriggerRatio 触发滑动的 Token 使用率阈值
    // 当 TokensUsed / MaxTokens >= 此值时触发批量滑动
    // 默认 0.65 (65%) — 在 "Lost in the Middle" 临界点之前动作
    SlideTriggerRatio float64

    // TargetRatio 滑动目标：滑动后希望达到的 Token 使用率
    // 默认 0.45 (45%) — 为后续 T-A-O 循环预留充足空间
    TargetRatio float64

    // MinPreserveMessages 滑动时最少保留的消息条数（防止过度裁剪）
    // 默认 4 (2 轮对话: user + assistant × 2)
    MinPreserveMessages int

    // MaxSlideBatch 单次滑动最多移除的消息条数上限
    // 默认 0 (不限制，由 Token 预算决定)
    MaxSlideBatch int
}

var DefaultSlideConfig = SlideConfig{
    SlideTriggerRatio:   0.65,
    TargetRatio:         0.45,
    MinPreserveMessages: 4,
    MaxSlideBatch:       0,
}
```

**为什么是 65% 触发 / 45% 目标？**

```
MaxTokens = 100% 的总预算分配：

┌─────────────────────────────────────────────────┐
│ System Prompt     ~5-10%  (固定开销)            │
│ Tool Definitions  ~5-10%  (固定开销)            │
│ User Input        ~3-5%   (每轮变化)            │
├─────────────────────────────────────────────────┤
│ Conversation History (ContextWindow 活跃区)      │
│   ← 滑动目标: 45% ← 触发点: 65% →             │
│                                                 │
│   45%-65% 区间 = 安全工作区                     │
│   >65%      = 触发滑动                          │
├─────────────────────────────────────────────────┤
│ Reserved for LLM Output ~25-30%                │
│   (Think + Act + Observation per iteration)     │
│   (T-A-O 可能跑 10-30 轮，每轮消耗不菲)         │
├─────────────────────────────────────────────────┤
│ Safety Buffer      ~5-10%  (紧急缓冲)           │
└─────────────────────────────────────────────────┘
```

### 决策 4：CurrentContext() 语义 — Token 预算感知

`CurrentContext(sessionID string, maxTokens int64) ([]*Message, error)` 

实现策略：从 SessionStore 的完整历史中，**从最新消息向前倒序选取**，直到 Token 总量 ≤ `maxTokens`。这确保返回的消息集始终在预算范围内。

### 决策 5：实现优先级

1. **Phase 1**: `MemorySessionStore`（纯内存，开发/测试用）
2. **Phase 2**: `BoltSessionStore`（bbolt 持久化，生产用）

### 决策 6：Agent 接入方式

- 新增 `WithSessionStore(store core.SessionStore)` Option
- `NewAgent()` 内部判断：若未设置则 Fallback 到 `MemorySessionStore`
- SessionStore 实例传递给 Reactor 持有和使用

## 二、接口设计

### 2.1 SessionStore 接口（增强版）

```go
// session.go

import "context"

// SlideEvent 当 ContextWindow 触发滑动时产生的回调事件
// 包含被滑出的消息列表，供消费者（如 RAG/Memory）进行语义化处理
type SlideEvent struct {
    SessionID string    `json:"session_id"`
    Slided    []Message `json:"slided"`     // 被滑出的消息（从 oldest 端移除）
    Remaining int       `json:"remaining"`   // ContextWindow 中剩余消息数
    Timestamp int64     `json:"timestamp"`   // 滑动发生时间
}

// SlideHandler 滑动事件的消费者回调函数类型
// 实现者可以在此将滑出的消息存入 RAG 或其他长期存储
type SlideHandler func(ctx context.Context, event SlideEvent)

// SessionStore 会话持久化存储接口
// 职责：有序地存储和恢复会话消息（WAL 模式）
// 不负责语义分析或知识提取 — 那是 Memory/RAG 的事
type SessionStore interface {
    // Append 将消息追加到指定会话尾部
    // System 角色的消息自动插入到头部区域
    Append(ctx context.Context, sessionID string, message Message) error

    // Get 获取指定会话的全部消息（完整历史）
    Get(ctx context.Context, sessionID string) ([]Message, error)

    // CurrentContext 根据 Token 预算获取适合当前上下文窗口的消息
    // 从最新消息倒序选取，直到总 Token 数 ≤ maxTokens
    // 这是 SessionStore 的核心方法 — 实现"滑动窗口"的读取侧
    CurrentContext(ctx context.Context, sessionID string, maxTokens int64) ([]Message, error)

    // Delete 删除指定时间戳的消息
    Delete(ctx context.Context, timestamp int64, sessionID string) error

    // Clear 清空指定会话的所有消息（用于会话重置）
    Clear(ctx context.Context, sessionID string) error

    // SetSlideHandler 注册滑动事件消费者
    // 当 ContextWindow 滑出消息时，通过此回调通知外部进行语义化处理
    SetSlideHandler(handler SlideHandler)

    // Close 关闭存储连接（资源清理）
    Close() error
}
```

## 三、核心流程

### 3.1 消息写入流程（每次 LLM 调用时）

```
1. 用户输入到达 → Reactor.Run(input, history)
                    ↓
2. contextWindow.AddMessage("user", input)
                    ↓
3. sessionStore.Append(sessionID, userMessage)     ← 持久化
                    ↓
4. 构建 LLM Prompt:
   history = sessionStore.CurrentContext(sessionID, availableTokens)
                    ↓
5. 调用 LLM → 得到 response
                    ↓
6. contextWindow.AddMessage("assistant", answer)
                    ↓
7. sessionStore.Append(sessionID, assistantMessage) ← 持久化
                    ↓
8. contextWindow.AddTokens(tokensUsed)
                    ↓
9. 【关键】检查是否需要滑动:
   if contextWindow.SlideTriggered(slideConfig) {
       slided := contextWindow.Slide(slideConfig, estimateFn)
       // 通过 SlideHandler 通知外部:
       sessionStore 的 handler(SlideEvent{Slided: slided.Messages})
       // 外部(Memory/RAG)消费 → 语义化 → 存入知识库
   }
```

### 3.2 滑动执行算法

```go
func (w *ContextWindow) Slide(config SlideConfig, estimateFn func(string) int) SlidedMessages {
    w.mu.Lock()
    defer w.mu.Unlock()

    estimate := estimateFn
    if estimate == nil {
        estimate = EstimateTokens
    }

    targetTokens := int64(float64(w.MaxTokens) * config.TargetRatio)
    var slided []Message
    var slidedTokens int64

    // 从最老端开始移除，直到总 Token ≤ 目标值
    // 同时保证最少保留 MinPreserveMessages 条
    for len(w.Messages) > config.MinPreserveMessages {
        totalTokens := w.calculateTotalTokens(estimate)
        if totalTokens <= targetTokens {
            break // 已达目标
        }

        removed := w.Messages[0]
        removedTokens := int64(estimate(removed.Content))

        w.Messages = w.Messages[1:]
        slided = append(slided, removed)
        slidedTokens += removedTokens

        if config.MaxSlideBatch > 0 && len(slided) >= config.MaxSlideBatch {
            break
        }
    }

    w.TokensUsed = w.calculateTotalTokens(estimate)
    w.LastActivityAt = time.Now()

    return SlidedMessages{Messages: slided, TokenCount: slidedTokens}
}
```

### 3.3 CurrentContext 实现策略

```go
func (s *MemorySessionStore) CurrentContext(_ context.Context, sessionID string, maxTokens int64) ([]Message, error) {
    s.mu.RLock()
    msgs, ok := s.store[sessionID]
    s.mu.RUnlock()

    if !ok || len(msgs) == 0 {
        return nil, nil
    }

    // 从最新消息倒序选取，直到 Token 预算耗尽
    var selected []Message
    var usedTokens int64
    for i := len(msgs) - 1; i >= 0; i-- {
        msgTokens := int64(EstimateTokens(msgs[i].Content))
        if usedTokens+msgTokens > maxTokens {
            break
        }
        selected = append([]Message{msgs[i]}, selected...) // 保持时间顺序
        usedTokens += msgTokens
    }
    return selected, nil
}
```

## 四、实现步骤（Task Breakdown）

### Phase 1: 基础设施

1. **增强 SessionStore 接口** (`core/session.go`)
   - 增加 `context.Context` 参数（所有方法）
   - 增加 `Clear()`, `Close()`, `SetSlideHandler()` 方法
   - 定义 `SlideEvent` 和 `SlideHandler` 类型

2. **定义 SlideConfig 和滑动算法** (`core/ctx_win.go`)
   - 新增 `SlideConfig` 结构体和默认值
   - 新增 `Slide()`, `SlideTriggered()`, `UsageRatio()` 方法
   - 新增 `SlidedMessages` 返回类型

3. **实现 MemorySessionStore** (`core/session_store_memory.go`)
   - 基于 `map[string][]Message` 的纯内存实现
   - 实现全部 SessionStore 接口方法
   - 线程安全（sync.RWMutex）

4. **新增 WithSessionStore Option** (`agent.go`)
   - `WithSessionStore(store core.SessionStore)` Option
   - `NewAgent()` 中 Fallback 到 `MemorySessionStore`

### Phase 2: Reactor 集成（核心重构）

5. **ContextWindow 下沉到 Reactor**
   - Reactor 新增 `sessionStore` 和 `contextWindow` 字段
   - 新增 `WithSessionStore` ReactorOption
   - Agent 创建 Reactor 时传入 SessionStore

6. **重构 Reactor.Run() 消息生命周期**
   - 每次 AddMessage 时同步 Append 到 SessionStore
   - 用 `sessionStore.CurrentContext()` 替代直接传 history
   - T-A-O 循环结束后检查并触发 Slide

7. **集成滑动事件通知**
   - SessionStore.SetSlideHandler 设置回调
   - 回调中可对接 Memory.Store() 进行语义化存储

8. **Agent 层简化**
   - 移除 Agent 中的 `contextWindow` 字段
   - 移除 Agent 中的 Prune/AddMessage 逻辑
   - Agent.Ask() 变为纯粹的 Reactor.Run() 代理

### Phase 3: 持久化实现

9. **实现 BoltSessionStore** (`core/session_store_bolt.go`)
   - 基于 `go.etcd.io/bbolt` 的文件持久化
   - Bucket 按 sessionID 隔离
   - 消息以 timestamp-ordered 存储

10. **测试覆盖**
    - MemorySessionStore 单元测试
    - Slide 算法边界测试（刚好触发、大量消息、空窗口等）
    - 端到端测试：多轮对话 + 自动滑动 + 无信息丢失验证

## 五、与现有机制的关系

### 与 ReNewer (Memory) 的关系

```
滑动窗口的两道防线：

第一道防线：SessionStore.Slide() （被动、机械式）
  ├─ 触发条件：Token 使用率 ≥ 65%
  ├─ 动作：批量移除 oldest 消息
  └─ 副作用：触发 SlideEvent → 可选地喂给 RAG

第二道防线：Memory.ReNewer.ReNew() （主动、语义式）
  ├─ 触发条件：maybeCompact 检测到超限
  ├─ 动作：从 Memory 中语义检索相关上下文重建
  └─ 位置：在 Reactor.maybeCompact() 中已有实现
```

两者**互补**而非冲突：
- **Slide** 解决的是"物理空间不够"的问题（机械式腾挪）
- **ReNew** 解决的是"语义质量下降"的问题（智能式重组）

理想情况下：Slide 先腾出物理空间 → 如果质量仍不够 → ReNew 做语义优化。

### 与 maybeCompact 的关系

当前的 `maybeCompact` 操作的是 Reactor 内部的 `ConversationHistory`（单次 Run 内）。新的滑动窗口机制操作的是跨 Run 的 `ContextWindow`。两者在不同粒度上工作：

| 维度     | maybeCompact (现有)                | Slide (新)                   |
| -------- | ---------------------------------- | ---------------------------- |
| 作用域   | 单次 Run() 内部                    | 跨多次 Ask()                 |
| 数据     | ConversationHistory                | ContextWindow / SessionStore |
| 触发频率 | 每个 T-A-O cycle 结束后检查        | 每次 Ask() 结束后检查        |
| 策略     | ReNew → MicroCompact → FullCompact | 批量移除 oldest → 通知 RAG   |

两者保留，互不干扰。
