# 记忆体设计（修订版 v2）

> 基于设计讨论后的共识整理。设计原则：Memory 是 ReActor 的"超频器"——无它可用，有它质变。

## 一、记忆体的定位

1. 用记忆体消除 LLM 的幻觉——通过 RAG 召回相关事实注入 prompt
2. 记忆体可以在团队间共享，形成团队经验（MemoryScope）
3. 记忆体的核心能力是语义化搜索。GoReact 不关心记忆体的实现，只提供通用接口。客户端可提供基于文本的简单实现，也可以提供基于 RAG 的完整实现
4. 经验记忆是 ReActor 的"生存线"——通过复用成功执行的分析结果，实现"越用越省"，将 Token 消耗降到最低

## 二、记忆类型体系

基于设计讨论，最终确定 5 种记忆类型：

| 枚举值 | 名称 | 含义 | 典型用途 |
|---|---|---|---|
| `MemoryTypeSession` | 临时记忆 | 当前会话上下文，不自动压缩也不清空 | 会话恢复、上下文重建 |
| `MemoryTypeUser` | 短时记忆 | 用户偏好设置 | 代码风格、语言偏好、操作习惯 |
| `MemoryTypeLongTerm` | 长期记忆 | 通用知识库（替代原 Project + Reference） | 架构决策、领域知识、项目规范 |
| `MemoryTypeRefactive` | 反射性记忆 | 工具/技能/代理的语义索引 | 语义化工具匹配 |
| `MemoryTypeExperience` | 经验记忆 | 成功执行路径的分析结果 | Skill 执行前的推理过程复用 |

**关于 SessionID**：MemoryRecord 不需要显式增加 SessionID 字段。Session 级别的隔离是客户端 Memory 实现的内部事务。ReActor 在调用 `ReNew` 时传入 sessionID，由客户端自行处理。

## 三、Memory 接口设计

### 3.1 核心接口

```go
type Memory interface {
    // Retrieve 搜索与 query 相关的记忆记录，按相关性排序
    Retrieve(ctx context.Context, query string, opts ...RetrieveOption) ([]MemoryRecord, error)

    // Store 持久化一条记忆记录，返回记录 ID
    Store(ctx context.Context, record MemoryRecord) (string, error)

    // Update 修改已有记录
    Update(ctx context.Context, id string, record MemoryRecord) error

    // Delete 删除记录
    Delete(ctx context.Context, id string) error

    // ReNew 上下文重建（可选方法，通过类型断言检测）
    // 当 ContextWindow 需要压缩时，由 ReActor 调用此方法
    // Memory 实现根据当前意图从临时记忆中语义召回相关上下文，返回精炼后的消息列表
    // 如果 Memory 不实现此方法，ReActor fallback 到原有的 LLM 摘要压缩
    ReNew(ctx context.Context, sessionID string, intent string, messages []Message) ([]Message, error)
}
```

**关于 ReNew 的设计决策**：
- ReNew 返回 `[]Message`（方案 A），由 ReActor 直接替换 ConversationHistory
- ReNew 是 Memory 接口的一部分，但使用类型断言检测是否可用（类似 Go 的 optional interface pattern）
- 如果 Memory 不实现 ReNew，ReActor fallback 到原有的 Compact 逻辑

**ReNew 调用时机**：
- 在 `maybeCompact` 中，当检测到上下文需要压缩且 `memory != nil` 时
- 优先调用 `ReNew`，失败时 fallback 到 `compactor.Compact` 或 `MicroCompact`

### 3.2 MemoryRecord 结构

```go
type MemoryRecord struct {
    ID        string       `json:"id"`
    Type      MemoryType   `json:"type"`
    Title     string       `json:"title"`
    Content   string       `json:"content"`
    Scope     MemoryScope  `json:"scope"`
    Tags      []string     `json:"tags,omitempty"`
    Score     float64      `json:"score,omitempty"` // relevance score from Retrieve
    CreatedAt time.Time    `json:"created_at"`
    UpdatedAt time.Time    `json:"updated_at"`
}
```

**不做扩展**：经验记忆的结构化数据（分析结果）序列化为 JSON 存入 `Content` 字段。ReActor 负责 serialization/deserialization，Memory 接口保持干净。

### 3.3 RetrieveOption

```go
type RetrieveConfig struct {
    Types    []MemoryType // 按类型过滤（空 = 全部）
    Scope    MemoryScope  // 按作用域过滤（0 = 不过滤）
    Limit    int          // 最大返回数量（0 = 默认 5）
    MinScore float64      // 最低相关性分数阈值（0 = 不过滤）
}
```

## 四、集成点设计

### 4.1 ContextWindow 与临时记忆：ReNew 替代 Compact

**当前架构**：`ContextWindow → LLM`，Compact = LLM 摘要压缩

**新架构**：ReActor 对会话只写不删。Compact 触发时：
- **无 Memory**：走原有路径（MicroCompact / Compactor.Compact）
- **有 Memory**：调用 `Memory.ReNew(sessionID, intent, messages)` → 得到 `[]Message` → 替换 ConversationHistory

```
// maybeCompact 中的分支逻辑
func (r *Reactor) maybeCompact(ctx *ReactContext) {
    // ... 计算 budget ...

    if budget.NeedCompact {
        if r.memory != nil {
            // 尝试 ReNew（开关式替代）
            if renewed, err := r.tryReNew(ctx); err == nil {
                ctx.ConversationHistory = renewed
                return
            }
        }
        // fallback: 原有 Compact 逻辑
        // ...
    }
}
```

**客户端侧的灵活性**：客户端收到 ReNew 请求后，可以从全部临时记忆中语义召回与当前意图相关的上下文 + 至少 5 轮近期对话，组合成新的 ContextWindow。客户端甚至可以将重要内容写入长期记忆。这些操作属于更高层次的业务处理，ReActor 无需关心。

### 4.2 反射性记忆：注册表的语义化 Fallback

**查询流程**：

```
精确匹配 (map[string] 查找)
    ↓ 未找到
语义化搜索 (memory.Retrieve(intent, WithMemoryTypes(MemoryTypeRefactive)))
    ↓ 得到 name 列表
从 map[string] 中查找对应的 Tool/Skill/Agent
```

**实现方式**：

1. 注册表（ToolRegistry、SkillRegistry）增加可选的 `memory core.Memory` 字段
2. 注册表增加 `SetMemory(mem core.Memory)` 方法
3. 查询逻辑增加 fallback 分支：精确匹配无结果时，调用 `memory.Retrieve` 获取名称列表，再精确查找
4. 反射性记忆的索引写入由客户端负责——ReActor 只需暴露注册表的只读访问（`All()`、`ListSkills()`），客户端自行决定何时、如何将工具/技能的语义描述写入 Memory

**ToolRegistry 改动**：
```go
type ToolRegistry struct {
    mu      sync.RWMutex
    tools   map[string]core.FuncTool
    memory  core.Memory  // 可选：注入后启用语义搜索
}

// SemanticSearch 尝试通过语义匹配查找工具
// 返回匹配的工具名称列表
func (r *ToolRegistry) SemanticSearch(query string) []string {
    r.mu.RLock()
    defer r.mu.RUnlock()
    if r.memory == nil {
        return nil
    }
    records, err := r.memory.Retrieve(
        context.Background(), query,
        core.WithMemoryTypes(core.MemoryTypeRefactive),
    )
    if err != nil {
        return nil
    }
    names := make([]string, 0, len(records))
    for _, rec := range records {
        if tool, ok := r.tools[rec.Title]; ok && tool != nil {
            names = append(names, rec.Title)
        }
    }
    return names
}
```

### 4.3 经验记忆：分析结果复用

**设计思路**：不在经验记忆中存"步骤序列"，而是存 Skill 成功执行前的**分析结果**（LLM 推理过程）。下次遇到类似任务时，将此"经验参考"传给 LLM，避免重复消耗分析 Token。

**经验的结构由两部分组成**：

| 部分 | 字段 | 说明 | 检索时的作用 |
|---|---|---|---|
| 问题/背景 | Title + Tags | 描述遇到的问题场景 | 作为召回的语义索引——用当前问题去匹配历史问题 |
| 解决办法 | Content | LLM 的分析推理过程 + 工具调用序列 | 作为参考注入 prompt，让 LLM 基于经验快速决策 |

**数据流**：
```
Skill 执行成功
    ↓
ReActor 将执行前的 Think 分析结果存入 Memory (Type=Experience)
    ↓
下次遇到类似任务
    ↓
Think 阶段 Retrieve 检索到相关经验（用"当前问题"匹配"历史问题"）
    ↓
经验作为"参考分析"注入 prompt
    ↓
LLM 基于经验快速决策，减少 Token 消耗
```

**存入时机**：在 T-A-O 循环中，当一个 Skill 被成功执行后（Observe 阶段确认成功），ReActor 将当前的 `Thought`（LLM 的分析推理）序列化为 JSON 存入经验记忆。

**经验记忆的 MemoryRecord 格式**：
```json
{
    "id": "exp_001",
    "type": 4,
    "title": "部署 Go 项目到远端服务器",
    "content": "{\"analysis\": \"需要先编译项目，然后通过 scp 传输到服务器...\", \"tool_sequence\": [\"bash\", \"bash\", \"ssh\"]}",
    "scope": 1,
    "tags": ["deployment", "go", "server"],
    "created_at": "2026-04-20T14:00:00Z"
}
```

> **扩展思考**：Memory 的实现完全可以将写入的经验生成为 Skill 文件（SKILL.md）保存到用户的 Skill 目录中。这样 ReActor + Memory 就构成一个**自成长系统**——Agent 越使用越强大，成功解决过的问题会自动转化为可复用的 Skill，不需要人工编写。这是 ReActor 与同类产品建立持久竞争壁垒的关键路径。

### 4.4 Think 阶段的记忆检索策略

Think 阶段从 Memory 中检索以下类型的记录：

1. **长期记忆**（MemoryTypeLongTerm）：与当前意图相关的项目知识、领域信息
2. **短时记忆**（MemoryTypeUser）：用户偏好、操作习惯
3. **经验记忆**（MemoryTypeExperience）：类似任务的历史分析结果

检索到的记录注入到 Think prompt 的 `<relevant_memory>` 区域。

## 五、所有权与注入模型

```
Agent（持有 Memory 实例）
  ↓ 注入
ReActor（通过 reactorSetup.memory 获取）
  ↓ 分发
  ├── Think 阶段：Retrieve 注入 prompt
  ├── Compact 阶段：ReNew 重建上下文
  ├── ToolRegistry：语义化搜索 fallback
  └── SkillRegistry：语义化搜索 fallback
```

- Memory 实例由客户端创建并注入 Agent
- Agent 通过 `reactor.NewReactor(config, WithMemory(mem))` 传递给 Reactor
- Reactor 内部各组件共享同一个 Memory 指针
